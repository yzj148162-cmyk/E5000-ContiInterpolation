# CDPR E5000 速度模式位置闭环控制设计

## 1. 文档目的

本文档定义一套基于雷赛 E5000 单轴连续速度运动接口的位置闭环测试方案。控制器以规划位置与实际位置的误差作为反馈，以规划速度作为前馈，通过 PID 计算实时速度命令，并使用以下库函数驱动电机：

- `dmc_vmove`：启动指定轴的连续速度运动；
- `dmc_change_speed_unit`：在运动过程中更新带符号的目标速度；
- `dmc_stop`：结束连续速度运动。

实施顺序为：先完成单轴闭环和跟随效果验证，再扩展到双轴，最后扩展到 8 轴。第一版不追求最短控制周期，先验证接口调用、方向切换、位置跟随、控制稳定性和安全停止是否可靠。

## 2. 总体原则

1. 新功能放在独立 Tab 页“速度闭环测试”中，避免操作入口和参数含义混淆。
2. 全局控制区继续在所有 Tab 页可见，包括控制卡初始化/关闭、轴使能/失能和全局立即停止。
3. UI 分页不能替代运动互锁。任何一种运动正在执行时，其他运动入口必须被禁用。
4. 所有 LTDMC 调用继续由独占硬件线程执行，UI、绘图和文件记录不得进入控制周期。
5. Trace 保持 1 ms 同帧采集指令位置和实际位置；第一版速度命令更新周期采用 10 ms。
6. 轨迹结束由位置误差、实际速度和连续稳定时间共同判断，不能只依赖规划时间到达终点。
7. 参数单位在上层统一使用物理角度：位置为 `deg`、速度为 `deg/s`、加速度为 `deg/s²`；进入库函数前再换算为 card unit。

## 3. 控制目标与控制律

### 3.1 输入信号

每个控制周期需要以下信号：

- 规划位置 `q_ref(t)`；
- 规划速度 `v_ref(t)`；
- Trace 实际位置 `q_act(t)`；
- Trace 实际速度 `v_act(t)`；
- 实际控制周期 `dt`。

单轴初始轨迹继续使用五次多项式。规划器应同时提供解析位置和解析速度，避免使用位置差分估计规划速度。每个控制周期根据本次运行的单调时钟计算当前轨迹时刻，再求取该时刻的 `q_ref(t)` 和 `v_ref(t)`；整条轨迹可以预先用于测试和绘图，但不能作为需要逐点写入板卡的硬件指令队列。

第一版每轴配置以下 Trace 对象：

| `dataType` | 对象 | 用途 |
|---:|---|---|
| 3 | 指令速度（整型） | 验证板卡内部实际采用的速度指令 |
| 4 | 反馈速度（整型） | PID 的 `v_act` 反馈 |
| 5 | 指令位置（整型） | 观察板卡内部累计的指令位置 |
| 6 | 反馈位置（整型） | PID 的 `q_act` 反馈 |

其中 `dataIndex=硬件轴号`、`dataSubIndex=0`、`slaveId=0`，整型对象先按4字节有符号数解析。首次开环试验必须用已知位置和速度核对原始值的单位、符号与脉冲当量换算关系；验证通过前不得把 type 4 直接投入PID。

### 3.2 位置误差

```text
e(k) = q_ref(k) - q_act(k)
```

积分项按真实控制周期更新：

```text
I(k) = I(k-1) + e(k) * dt
```

微分项不使用 `e/dt`。优先使用：

```text
D(k) = v_ref(k) - v_act(k)
```

该写法等价于误差变化率 `de/dt`，同时可以直接利用解析规划速度和 Trace 实际速度。

上述 `e(k)` 是轨迹时间误差：它保留系统响应延迟，用于 PID 和终点稳态判定，
判断电机是否在规定时刻到达规划位置。Trace 延迟标定通过后，诊断、曲线和
最大跟随误差保护同时使用延迟对齐跟随误差：

```text
e_aligned(t) = q_cmd_type05(t - tau) - q_act_type06(t)
```

其中 `tau` 取当前轴有效标定结果；标定无效时使用程序明确显示的默认值。该误差
排除固定响应延迟，用于评价板卡指令执行后的空间跟随精度，并作为运行期最大
跟随误差保护的判定量，但不替换 PID 误差和终点绝对位置误差。若运动过程中
延迟对齐误差因 Trace 历史帧不连续而持续无效，超过 Trace 超时时间后按反馈异常
停止。批量读取多帧 Trace 时，每一帧必须按自身卡侧时间戳查询和线性插值历史
type05，不能用批次最后一帧代替；最大跟随误差保护检查本批全部有效帧的绝对
峰值，避免漏掉批次中间的瞬时超限。

### 3.3 PID 与速度前馈

```text
v_raw = v_ref
      + Kp * e
      + Ki * I
      + Kd * (v_ref - v_act)
```

各项单位关系如下：

| 参数 | 建议单位 |
|---|---|
| `Kp` | `1/s` |
| `Ki` | `1/s²` |
| `Kd` | 无量纲 |
| PID 输出 | `deg/s` |

第一轮调试从前馈开始，随后依次增加 P、I、D，避免一次引入过多变量。

### 3.4 输出限制

先限制最大速度：

```text
v_limited = clamp(v_raw, -v_max, +v_max)
```

再限制每周期速度变化量：

```text
delta_v_max = a_max * dt
delta_v = clamp(v_limited - v_cmd_previous,
                -delta_v_max,
                +delta_v_max)
v_cmd = v_cmd_previous + delta_v
```

速度限幅防止超过电机和机构允许速度；速度变化率限幅防止 PID 因瞬时位置误差输出剧烈速度跳变。

### 3.5 积分抗饱和

积分项必须设置上下限。当速度命令已经饱和且误差仍推动输出继续向饱和方向增大时，暂停积分；当误差有助于退出饱和时允许积分恢复。

第一版默认 `Ki=0`，待 P 控制稳定后再启用积分。

### 3.6 参考量产生与速度命令交付

本方案与轨迹缓冲式执行有本质区别：上位机不向板卡逐点写入位置，也不存在 mark、预压、前瞻或补段。每个控制周期只完成一次闭环计算：

```text
按实际轨迹时刻计算 q_ref、v_ref
        ↓
读取最新完整 Trace 帧的 q_act、v_act
        ↓
计算并限幅得到当前 v_cmd
        ↓
将 v_cmd 交给硬件线程在线更新轴速度
```

控制逻辑线程与硬件线程之间采用“最新值邮箱”，每条命令至少包含运行序号、轴号、生成时间戳、速度值和有效标志。新的 `v_cmd` 直接覆盖尚未执行的旧值，硬件线程只执行当前运行中最新且未过期的命令，禁止使用无限增长的 FIFO 依次补发历史速度。若速度命令超过允许时限仍未被硬件线程处理，应报告控制超时并进入安全停止，不能事后追赶执行。

## 4. 雷赛库函数调用策略

### 4.1 方向与在线变速

手册定义：

- `dmc_vmove(card, axis, 0)`：负方向连续运动；
- `dmc_vmove(card, axis, 1)`：正方向连续运动；
- 连续速度运动中，`dmc_change_speed_unit` 的 `New_Vel` 可以为负值或正值，分别表示负向和正向速度。

因此，运动启动后可通过带符号速度命令完成减速、跨零和反向，不需要每次改变方向都停止并重新调用 `dmc_vmove`。反向能力仍需在单轴低速试验中单独验证。

### 4.2 在线变速时间

函数原型：

```cpp
dmc_change_speed_unit(card, axis, newVelocity, Taccdec);
```

`Taccdec` 表示板卡从当前速度变到新速度所用的时间。第一版采用：

```text
控制周期 = 10 ms
Taccdec   = 10 ms
```

即板卡在一个控制周期内过渡到本周期的新速度命令。UI 中保留“在线变速时间”参数，便于对比 0、10、20、50 ms 的响应与滞后。

注意：手册说明在线变速成立后，板卡会重新计算并覆盖默认速度、加减速时间。每次开始新试验前必须重新调用 `dmc_set_profile_unit` 和 `dmc_set_s_profile`，不能沿用上一次运行后的回读参数。

### 4.3 S 曲线

手册建议 S 形速度曲线最好在恒速段进行在线变速，而本方案会周期性更新速度。第一版将 `s_para` 设为 0，不额外启用 S 曲线；运动平滑性由软件速度变化率限制和 `Taccdec` 共同保证。

### 4.4 启动状态机

五次多项式起点速度为 0，不能一开始就以最大速度调用 `dmc_vmove`。启动过程定义为：

1. 检查控制卡、总线、轴状态机、Trace 和软件互锁；
2. 读取 Trace 实际位置作为本次规划起点；
3. 重置 PID、积分项、控制时钟和数据记录；
4. 当 `|v_cmd|` 首次超过启动速度阈值时，按符号确定 `dmc_vmove` 方向；
5. 使用当前安全速度配置单轴速度曲线并调用一次 `dmc_vmove`；
6. 后续周期只调用 `dmc_change_speed_unit` 更新带符号速度。

启动速度阈值用于避免极小速度命令因编码器量化和数值噪声造成方向反复切换。

### 4.5 正常结束

规划时间到达末端后，保持：

```text
q_ref = 最终目标位置
v_ref = 0
```

PID 继续输出小速度修正。满足以下条件并连续稳定一段时间后结束：

```text
|q_ref - q_act| <= 位置容差
|v_act|         <= 速度容差
持续时间        >= 稳态确认时间
```

结束顺序：

1. 将速度命令限制到 0；
2. 确认实际速度接近 0；
3. 调用 `dmc_stop(..., 0)`；
4. 停止控制定时器和运行日志；
5. 保留最终曲线和数据文件；
6. 根据用户选择决定是否自动失能轴。

### 4.6 异常停止

以下任一情况触发安全停止：

- Trace 超时或连续无有效帧；
- 轴状态机不再为操作使能状态；
- EtherCAT 总线错误；
- 位置跟随误差超过安全阈值；
- 实际速度超过安全阈值；
- `dmc_change_speed_unit` 连续调用失败；
- 控制周期连续多次严重超时；
- 用户点击全局立即停止；
- 程序关闭。

异常时优先停止速度命令和运动，再执行轴失能及控制卡关闭。

## 5. 软件架构

### 5.1 模块划分

建议新增以下独立类型：

```text
VelocityControlConfig
    单轴控制参数、PID参数和安全阈值

VelocityControlStatus
    当前参考量、反馈量、PID输出、API结果和运行状态

PositionVelocityPid
    纯计算类，不调用Qt和LTDMC，便于离线测试

VelocityControlWorker
    轨迹状态机、控制周期、结束判定和日志组织

E5000HardwareInterface扩展
    配置速度曲线、启动vmove、在线变速、停止及Trace读取
```

旧执行链保持冻结，新速度闭环状态机不得复用旧运行标志或定时器，避免两套流程互相改变硬件状态。

### 5.2 线程边界

```text
UI线程
  参数输入、按钮、状态显示、低频绘图
        ↓
控制逻辑线程
  轨迹时间、PID、结束判定、产生轻量结构化日志事件
        ↓
独占硬件线程
  Trace读取、dmc_vmove、dmc_change_speed_unit、dmc_stop
        ↓
记录线程
  控制数据及运行日志批量写文件
```

第一版控制周期为 10 ms。必须记录真实 `dt`、最大周期抖动和库函数耗时。确认 10 ms 稳定后，再评估 5 ms 或更短周期；不能直接把 Qt 定时器等同于 EtherCAT 硬实时周期。

轨迹参考量和PID由控制逻辑线程计算，硬件线程只接收最新速度命令并访问LTDMC。硬件线程不得因为控制线程短时产生多条命令而顺序重放旧速度；每次调用 `dmc_change_speed_unit` 前必须再次核对运行序号、命令时效和运动所有权。

日志事件可以由控制逻辑线程判定和产生，但该线程只允许构造包含事件类型、时间戳、轴号、错误码及少量数值的轻量结构体，并将其放入有容量上限的异步队列。控制逻辑线程不得直接执行以下操作：

- 调用 `appendPlainText()` 等UI文本操作；
- 进行大量 `QString::arg()` 格式化；
- 写入CSV、文本或二进制文件；
- 因等待日志队列、UI线程或磁盘而阻塞控制循环。

UI线程按 100～200 ms 周期批量取出事件，完成字符串格式化和日志显示；记录线程负责异步批量落盘。队列满时优先保留错误、急停和状态切换事件，允许合并或丢弃重复的普通诊断事件，不能反向阻塞PID控制。

### 5.3 模式互锁

应用内设置唯一的运动所有者：

```text
Idle
PointMove
VelocityClosedLoop
Stopping
Error
```

只有 `Idle` 状态允许启动新的运动。切换 Tab 不改变运动所有权，也不能绕过停止流程。

### 6.5 曲线

至少绘制：

1. 位置跟踪图：`q_ref`、Trace type 5 指令位置和 type 6 实际位置；
2. 位置误差图：同时显示轨迹时间误差 `q_ref(t)-q_act(t)`、延迟对齐误差
   `q_cmd_type05(t-tau)-q_act_type06(t)`以及轨迹时间误差的正负允许范围；

### 7.2 运行日志

普通状态每 250 ms 输出一次，不按 10 ms 刷屏。以下事件立即记录：

- 运行参数快照；
- `dmc_vmove` 启动方向和返回码；
- 第一次在线速度命令；
- 速度跨零和反向；
- PID进入/退出饱和；
- 跟随误差告警；
- API失败或控制周期超时；
- 进入终点稳态和最终结束；
- 用户停止和异常停止原因。

“立即记录”表示控制线程立即把结构化事件送入异步队列，不表示立即操作UI或同步写文件。运行日志与每周期控制数据使用独立队列：前者按事件显示，后者只进入记录线程和低频曲线缓存，避免高频PID样本被误当作文本日志处理。

UI日志刷新采用批量追加，并设置最大保留行数；超过上限时删除最早的普通日志，防止长时间运行导致文本控件持续膨胀。记录线程应批量写入并定期刷新文件，停止运行时完成剩余队列的有限时收尾；异常停机不得为了等待日志落盘而延迟硬件停止。

## 8. 单轴调试流程

### 8.1 阶段 A：速度接口开环验证

不运行位置 PID，依次验证：

1. 正向低速 `dmc_vmove`；
2. 在线阶跃变速；
3. 命令速度降到 0；
4. 正速度跨零变为负速度；
5. 不同 `Taccdec` 的响应；
6. 用已知速度核对 type 3、4 原始值的单位、符号和脉冲当量换算；
7. 100、50、20、10 ms 调用周期下的返回码、耗时和实际速度响应。

只有 10 ms 在线变速稳定后，才进入位置闭环试验。

### 8.2 阶段 B：仅速度前馈

```text
Kp = 0
Ki = 0
Kd = 0
v_cmd = v_ref
```

观察位置误差、速度滞后、方向切换和API响应，建立硬件通道基线。

### 8.3 阶段 C：前馈加 P

从很小的 `Kp` 开始逐步增加，观察：

- 最大位置误差；
- 稳态误差；
- 超调；
- 速度命令是否频繁饱和；
- 实际位置是否产生振荡。

### 8.4 阶段 D：加入 I

在 P 控制无振荡后增加小 `Ki`，用于消除持续偏差。必须同时验证积分限幅和饱和冻结。

### 8.5 阶段 E：评估 D

如果 P/PI 存在明显超调或阻尼不足，再逐步增加 `Kd`。实际速度含编码器量化噪声，D 项需要重点观察速度噪声对命令的放大。

### 8.6 单轴通过条件

- 正向、减速、停止和反向行为均符合命令；
- 10 ms 控制周期下API无持续错误；
- 控制周期和API耗时满足预算；
- 位置误差的最大值、均方根和终点值可重复；
- 命令速度不长期饱和；
- 轨迹结束后日志和曲线正确停止；
- 主动停止、异常停止和程序关闭均可安全结束运动。

## 9. 多轴扩展

### 9.1 双轴

双轴必须使用同一个单调时钟和同一个控制周期：

1. 同一时刻计算两轴 `q_ref/v_ref`；
2. 从同一 Trace 帧取得两轴实际位置；
3. 完成两轴 PID 计算；
4. 在独占硬件线程中连续发送两次 `dmc_change_speed_unit`；
5. 记录两次API调用的起止时间和发送时间差。

双轴验收除了各轴误差，还需要记录轴间相位差和到达时间差。

### 9.2 8 轴

8轴扩展不改变控制律，只扩展数组维度和安全检查。每周期必须先完成全部轴计算，再统一进入发送阶段，不能边读取边计算边发送。

由于 `dmc_change_speed_unit` 是单轴接口，8次调用不是硬件原子同时执行。必须测量：

```text
第一轴命令发送时刻
最后一轴命令发送时刻
单周期8轴总API耗时
各轴Trace响应相位差
```

只有发送时间差和轴间同步误差满足系统要求，才继续把该方式用于8轴实验。
