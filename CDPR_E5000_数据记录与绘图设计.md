# CDPR E5000 数据记录、绘图与离线轨迹评估设计

## 1. 目的与适用范围

本文定义 CDPR E5000 连续插补测试后续的数据采集、记录、实时显示和离线评估方案。

核心目标不是在高频控制周期内直接绘图或写文本文件，而是完整、可追溯地保留 8 绳实际位置的时间历程，并据此离线计算末端正运动学轨迹，最终判断规划轨迹、控制卡指令轨迹与实际执行轨迹之间的差异。

本文服务于未来 8 轴连续插补测试；当前两轴阶段可按相同接口逐步接入。本文不改变 E5000 连续插补本身的喂点和速度倍率逻辑，二者的时间关联见《CDPR_E5000_8轴插补时间同步与速度倍率控制设计》。

## 2. 设计原则

1. **控制优先。** 任何记录、绘图、导出操作都不得阻塞硬件线程、插补喂点线程或安全逻辑。
2. **原始数据优先。** 高速记录保存脉冲原值及必要元数据；显示用单位值、滤波值和正运动学结果可重新计算。
3. **同帧优先。** 8 轴位置必须来自同一 Trace 帧，不能将 8 次顺序读取的“最新值”拼成一个伪同步样本。
4. **时间可追溯。** 用 Trace 帧序号和已配置采样周期重建采样时间；上位机收到数据的时刻仅作为诊断辅助。
5. **实时显示降采样。** UI 只显示最新快照或降采样曲线，不承担原始高频数据的保存职责。
6. **故障可说明。** 缺帧、溢出、写盘延迟、Trace 错误和正运动学不收敛都必须被记录，而不是静默丢失。

## 3. 总体数据通路

```text
E5000 Trace PDO（同一帧：8轴指令位置 + 8轴实际位置）
                         ↓
E5000HardwareInterface 硬件线程
  读取并校验 Trace 帧、附加帧序号/时间基准/运行状态
                         ↓
固定容量单生产者单消费者环形队列（Telemetry Ring Buffer）
                  ↓                         ↓
记录线程 TelemetryWriter                 UI 快照通道
  批量写入二进制数据文件                     20～50 Hz
                  ↓                         ↓
离线转换/分析工具                         曲线与状态显示
                  ↓
正运动学、规划对齐、误差统计、报告导出
```

硬件线程是唯一可调用雷赛库函数的线程。Trace 读取、连续插补状态读取和运行状态标记均在该线程完成；记录线程和 UI 线程只消费复制后的普通数据结构，绝不反向访问控制卡。

## 4. 三层数据模型

为了区分“规划问题、插补执行问题和机械跟随问题”，一次试验至少记录以下三层数据。

| 层级 | 主要内容 | 用途 |
|---|---|---|
| 规划层 | 规划时间、轨迹点/mark、8 轴规划目标、末端规划位姿、规划速度、轨迹段编号 | 定义期望时间历程 |
| 插补层 | 已推送 mark、当前消费 mark、连续插补缓冲余量、公共速度倍率及其分量、喂点结果 | 判断时间同步和断粮风险 |
| Trace 执行层 | 同帧 8 轴指令位置、8 轴实际位置、伺服状态、轴错误码、Trace 状态 | 判断控制卡输出和电机实际跟随 |

离线正运动学的主要输入是 Trace 实际位置。Trace 指令位置用于区分两类误差：

```text
规划轴位置 − Trace 指令位置  →  插补/时间进度误差
Trace 指令位置 − Trace 实际位置 →  伺服跟随误差
Trace 实际位置 → 绳长 → 正运动学位姿 → 末端执行误差
```

## 5. 时间基准与对齐

### 5.1 Trace 时间

配置 Trace 时，记录实际生效的采样周期 `tracePeriodUs`（例如 500 us 或 1 ms）。每个有效 Trace 帧带有单调递增的 `traceSequence`。一次运行的采样相对时间定义为：

```text
t_trace = (traceSequence - firstTraceSequence) × tracePeriodUs
```

若控制卡 API 直接提供硬件时间戳，可以额外保存，但仍保留上述帧序号和周期，以便复核。不得以 Qt 定时器触发时间或“批量读取到该帧的时刻”替代采样时间。

### 5.2 规划时间

规划层中的每个点必须保存 `mark`、规划相对时间 `t_plan` 和 8 轴规划目标。运行开始时保存 `tExecuteStart`、起始 mark 和轨迹版本号。离线分析以此建立 `t_plan` 与 `t_trace` 的对应关系。

### 5.3 延迟、缺帧与分段

下列事件必须作为独立事件记录：开始/停止、暂停、急停、使能变化、速度倍率突变、Trace API 错误、帧序号跳变、环形队列溢出、文件写入失败。

发生缺帧或试验中断时，离线工具必须将数据切分为多个连续区间，禁止跨缺口直接插值并给出“连续误差”。

## 6. 记录数据格式

### 6.1 高频帧记录

高频路径使用固定大小的二进制帧，避免每一帧创建 `QJsonObject`、`QString` 或 CSV 文本。建议的逻辑字段如下：

```text
TelemetryFrame
  runId, traceSequence, tTraceNs
  validAxisMask, traceStatus, boardError, eventFlags
  commandPulse[8], actualPulse[8]
  axisState[8], axisError[8]
  currentMark, pushedMark, contiBufferedLines
  ratioCmd, ratioRef, ratioPhase, ratioBuffer
```

位置原值优先保存为整数脉冲。`1000 pulse/deg`、`360000 pulse/rev`、轴方向、绳长零点和传动参数等不能只写在代码中，而应作为本次运行的元数据写入文件头。

力传感器、速度、电流等后续量可使用独立的 Trace 帧类型或独立数据流；必须包含自身的采样周期和时间基准，不能默认与位置帧完全同时。

### 6.2 元数据与文件组织

每次试验建立唯一目录，例如：

```text
run_20260715_153012/
  metadata.json
  trace_position.bin
  planner.bin
  events.bin
  export/                 （离线生成的 CSV、图和报告）
```

`metadata.json` 只在开始和结束阶段写入，至少包括：软件版本/提交号、控制卡与驱动器信息、Trace PDO 配置、采样周期、参与轴映射、单位与方向、零位/绳长标定版本、轨迹文件或参数、速度倍率参数、开始结束原因和丢帧统计。

高频数据的离线导出可生成 CSV 或表格文件；CSV 不作为实时记录格式。

## 7. 非阻塞记录机制

### 7.1 环形队列与记录线程

硬件线程是单一生产者，`TelemetryWriter` 是单一消费者。两者通过预分配、固定容量的 SPSC 环形队列传递 `TelemetryFrame`。

- 硬件线程只进行固定大小结构体拷贝和原子索引推进；不做文件打开、格式化、动态分配或 `flush()`。
- 记录线程每次批量取出若干帧，顺序写入缓冲文件；按固定批次或约 100～500 ms 刷新一次操作系统缓冲。
- 正常停止时先停止生产，再排空队列、写入结束元数据并关闭文件。
- 记录线程异常、磁盘空间不足或文件打开失败时，通知控制状态机；是否继续运动由后续安全策略明确规定，不能悄然忽略。

### 7.2 溢出策略

记录队列满时，硬件控制和连续插补仍优先执行。当前方案建议：丢弃新到的遥测帧、原子增加 `droppedTelemetryFrames`、立刻记录溢出事件，并在 UI 明确报警。不得为了保证记录完整性而阻塞硬件线程。

正式精度试验应将任何溢出视为数据无效；可在试验结束后标记“不可用于完整轨迹评估”。队列容量应按照最长可接受写盘阻塞时间估算，并在试验前检查磁盘剩余空间。

## 8. 实时 UI 与绘图

原工程的思路可以沿用：高频采集与低频界面解耦。建议 UI 快照刷新频率为 20～50 Hz，曲线重绘频率约 20～50 Hz；显示的是“当前最新样本”，而非让 UI 消费每一个 Trace 帧。

实时曲线建议提供：

1. 8 轴指令/实际位置及跟随误差；
2. 连续插补缓冲余量、`currentMark`、`pushedMark`、公共速度倍率；
3. 可选的绳力、速度和轴状态；
4. 记录队列占用率、丢帧数、Trace 错误数和磁盘写入状态。

绘图窗口只保留最近固定时长（例如 10～30 s）的显示数据，并对超高频数据做抽取或 min/max 包络降采样。调用 QCustomPlot 等绘图库的 `replot()` 必须在 UI 线程，以排队方式触发；不得从硬件线程直接绘图。

## 9. 离线正运动学与评估流程

离线分析按以下顺序进行：

1. 读取元数据，校验轴映射、单位、方向、零点和采样周期；
2. 以 `traceSequence` 重建 Trace 时间轴，识别缺帧与无效区段；
3. 将实际脉冲转换为各绳实际长度；保存转换后的长度而不覆盖原始脉冲；
4. 对每一个连续数据区段进行 CDPR 正运动学，保存求解位姿、收敛状态、迭代次数和残差；
5. 以规划时间、mark 和经校准的执行延迟对齐规划位姿和实际位姿；
6. 分别统计轴空间误差、绳长误差、末端位置误差、姿态误差、时间滞后和最大/均方根误差。

正运动学“不收敛”或残差过大时，该样本必须明确标记为无效，不能将上一次成功位姿伪装成当前结果。离线报告应同时给出有效样本比例与残差分布。

## 10. 规模估算与参数建议

以 8 轴、每轴指令/实际各一个 32 位整数、1 ms 采样为例，位置原始数据量约为 64 KB/s；加入状态、mark、倍率和文件头后通常仍在数百 KB/s 量级。若采用 500 us 采样，数据量约翻倍。实际容量以最终 `TelemetryFrame` 字节数计算，并预留至少一小时试验空间和文件轮转策略。

第一版建议参数：

| 参数 | 建议初值 | 说明 |
|---|---:|---|
| Trace 位置采样周期 | 1 ms 或 500 us | 以控制卡稳定读取能力为准 |
| UI 快照/重绘 | 20～50 Hz | 与 Trace 采样解耦 |
| 记录写盘批次 | 100～500 ms | 不在硬件线程写盘 |
| 显示窗口 | 10～30 s | 固定长度、滚动显示 |
| 队列容量 | 至少覆盖数秒 | 以写盘抖动测试结果定标 |

## 11. 分阶段实施

### 阶段 A：两轴基础能力

- 复用现有 Trace 指令/实际位置读取链路；
- 定义 `TelemetryFrame`、元数据和事件格式；
- 建立环形队列与独立记录线程；
- 记录两轴数据、Trace 状态和丢帧计数，提供低频位置/误差曲线。

### 阶段 B：8 轴连续插补

- 同帧读取 8 轴 Trace 数据；
- 接入规划 mark、缓冲余量和三分量速度倍率；
- 保存完整的规划层、插补层和执行层数据；
- 增加记录状态与磁盘空间的试验前检查。

### 阶段 C：离线评估工具

- 二进制数据转换与完整性校验；
- 绳长转换、CDPR 正运动学与残差诊断；
- 生成规划/指令/实际轨迹对比图、误差统计和可复现实验报告。

## 12. 验收标准

一次可用于插补效果评估的试验至少满足：

1. 8 轴有效数据来自同一 Trace 帧，且全程无未说明的帧序号跳变；
2. 记录队列无溢出，文件关闭完整，元数据齐全；
3. 可重建规划、插补和 Trace 三层时间轴；
4. 可分别绘制规划、Trace 指令和 Trace 实际轴位置；
5. 可由 Trace 实际位置离线得到绳长和末端位姿，并输出正运动学收敛/残差；
6. 任何缺帧、急停、错误、暂停或参数变更均可在事件记录中定位。
