# CDPR E5000 连续插补顿挫排查与优化设计

## 1. 目的与适用范围

本文用于排查 `conti_ctrl` 当前两轴阶段一、阶段二连续插补中出现的“运动断断续续、速度反复掉落或在段端停住”现象，并定义后续优化顺序。

本文的原则是：**先通过可复现的对照试验定位，再一次只修改一个变量**。不得在尚未区分“卡侧断粮”“小线段规划”“倍率控制”“驱动侧保护”之前，同时调整多个参数后仅凭肉眼判断结果。

本文不替代以下设计文件：

- 《CDPR_E5000_连续插补方案》：连续插补测试总体方案；
- 《CDPR_E5000_8轴插补时间同步与速度倍率控制设计》：未来八轴时间同步与公共倍率方案；
- 《CDPR_E5000_数据记录与绘图设计》：Trace 数据记录与离线评估方案。

---

## 2. 当前程序执行链路

当前两轴测试的轨迹与喂点链路为：

```text
五次多项式轨迹
  每 10 ms 生成一个两轴绝对位置点
          ↓
上位机轨迹队列（hostQueue）
          ↓
启动前预压：默认覆盖约 200 ms 的有效线段
          ↓
dmc_conti_open_list + dmc_conti_line_unit + dmc_conti_start_list
          ↓
运行中每 5 ms 检查 mark、卡侧空槽、上位机队列并尝试补段
          ↓
控制卡连续插补缓冲区、前瞻、矢量速度/S 曲线、公共速度倍率
          ↓
EtherCAT 伺服 + Trace 指令位置/实际位置反馈
```

程序运行期间还存在两条并行但同属 `ContiWorker` 事件循环的定时任务：

- 轨迹产点：10 ms；
- Trace 读取与反馈刷新：10 ms；
- 卡侧喂点与倍率控制：5 ms。

`E5000HardwareInterface` 独占硬件线程，雷赛库函数不会由 UI 线程直接调用；但是 `ContiWorker` 对硬件线程的调用是同步等待的。因此 Trace 读取、状态读取或压入线段耗时过长时，仍可能推迟同一事件循环中的后续产点和喂点任务。

## 3. 先明确三个概念

### 3.1 `remainSpace` 是空槽数，不是剩余轨迹量

`dmc_conti_remain_space` 返回连续插补缓冲区中**还可以写入多少段**。

```text
remainSpace 很大（例如接近 5000） → 卡侧已排队的段很少，甚至为空
remainSpace 很小                   → 卡侧已排队的段较多，接近容量上限
```

因此，“缓冲剩余空间很大”不能视为安全；对连续运动而言，它反而可能表示卡侧即将断粮。

### 3.2 `currentMark / pushedMark` 是段进度，不是精确时间

`currentMark` 来自 `dmc_conti_read_current_mark`，表示控制卡已执行到的连续插补段进度；`pushedMark` 是程序已成功压入的最后一个 mark。

```text
待执行段数近似值 = pushedMark - currentMark
```

该值适合判断是否接近断粮，但 mark 的精确边界语义必须通过实机单段试验校准，不能将一个 mark 的偏差直接等同为固定时间误差。

### 3.3 “停止”与“顿挫”不是同一问题

- **停止**：控制卡或伺服明确进入停止、暂停、报警或失能状态；
- **顿挫**：电机仍在连续执行，但每个小段之间反复减速、倍率反复降低，或短暂等待下一批线段。

两者需要用 `runState`、mark、缓冲量、倍率和 Trace 曲线共同区分，不能只靠肉眼判断。

## 4. 当前代码中明确会停止运动的路径

当前程序没有调用 `dmc_conti_pause_list`、`dmc_conti_delay`。`dmc_conti_change_speed_ratio` 用于在线调整倍率，不是暂停接口。

但以下路径会调用 `dmc_conti_stop_list`、关闭连续插补列表，或使运动自然结束：

| 类别 | 触发条件 | 预期日志/状态 | 是否属于故障 |
|---|---|---|---|
| 用户操作 | 点击减速停止、全局立即停止、安全关闭控制卡、退出程序 | “减速停止”“立即停止”“安全停机” | 否 |
| 正常结束 | 轨迹生产结束、所有段已消费 | “五次多项式轨迹已执行完成” | 否 |
| 库函数错误 | 开表、压入线段、启动、倍率调整等调用失败 | “错误：...” | 是 |
| Trace 错误 | 连续插补运行期间 Trace 读取失败 | “运行中位置 Trace 读取失败” | 是 |
| 首段未执行 | 启动后超时，`currentMark` 未进入首个有效段 | “仍未消费首个有效段” | 是 |
| 驱动/总线侧 | 伺服报警、限位、急停链路、EtherCAT 异常 | 轴错误码、状态机、总线状态变化 | 是 |

`dmc_conti_get_run_state` 的常用含义如下：

| runState | 含义 | 当前排查含义 |
|---:|---|---|
| 0 | 运行 | 控制卡正在消费连续插补段 |
| 1 | 暂停 | 应检查是否存在外部暂停或库函数调用 |
| 2 | 正常停止 | 可能是轨迹自然结束，也可能是卡侧段已经耗尽 |
| 3 | 未启动 | 启动流程或状态异常 |
| 4 | 空闲 | 列表关闭、错误停止或未运行 |

特别注意：目前程序只将 `3/4` 直接视为“意外停止”。若轨迹尚未生产完成却出现 `runState=2`，应视为“疑似断粮/提前结束”的重点诊断事件；后续需要把这一状态显式记录并处理。

## 5. 顿挫原因清单与判据

### 5.1 卡侧断粮或近断粮

**机理：** 控制卡完成已排入的最后一条线段，但后续轨迹尚未成功压入。卡只能在最后目标点减速、停住或等待，之后即使又有新点，也会形成可见的速度断裂。

**可能诱因：**

- 轨迹产点速度低于卡侧消费速度；
- 预压时间过短；
- 喂点定时器延迟；
- Trace 读取、日志或同步硬件调用挤占 `ContiWorker` 事件循环；
- 当前倍率高于可持续的供点速度；
- 产点因脉冲量化被跳过，导致有效段覆盖时间不足。

**诊断特征：**

```text
currentMark 接近 pushedMark
缓冲段数趋近 0
remainSpace 迅速增大、接近卡侧总容量
hostQueue 也可能接近 0
runState 可能由 0 变为 2
```

**首要对照试验：** 在阶段一把整条短轨迹一次性预装入卡侧（不超过卡侧段容量），并关闭时间同步倍率控制。若顿挫消失，问题位于流式产点/喂点或其调度，而不是电机和连续插补基本功能。

### 5.2 小线段没有被前瞻平滑融合

**机理：** 当前每 10 ms 生成一条 `dmc_conti_line_unit` 小线段。若控制卡没有将相邻段充分前瞻融合，就可能在每一段端点按矢量加减速规划减速，再进入下一段加速。

**当前风险：** 已启用小线段前瞻，但 `PathError=0`。对于完全共线的阶段一轨迹，理论上不需要拐角容差；但是否实际融合、需要多少前瞻段，必须实机验证，不能仅以“已调用设置函数”判定有效。

**诊断特征：**

```text
mark 持续前进
hostQueue 有余量，pushedMark - currentMark 始终较大
remainSpace 未表现为接近空队列
倍率基本稳定
但 Trace 实际位置的速度呈周期性锯齿或反复接近零
```

**首要对照试验：** 以相同起终点分别执行：

1. 一条长直线段；
2. 由当前 10 ms 小段组成的同一共线轨迹。

若只有第二种顿挫，应优先检查前瞻配置、可用前瞻段数、轨迹离散周期和矢量加减速参数。

### 5.3 公共速度倍率被压低或频繁变化

**机理：** 时间同步控制为防止断粮，会在低水位时把倍率压向下限；相位误差、参考速度和缓冲修正也会共同改变倍率。倍率不应命令零速，但若反复变化，仍会造成明显的加减速感。

**当前边界：** 倍率下限大于零时，倍率控制本身不应主动让电机完全静止；但它可能通过降低/提高消费速度，掩盖或放大供点不足问题。

**诊断特征：**

```text
“在线速度倍率已生效”日志与顿挫时间重合
ratioApplied 在较短时间内反复变化，或长期贴近 ratioMin
bufferTimeMs 经常低于 lowBufferMs / criticalBufferMs
```

**对照试验：** 必须实现真正的“静态倍率模式”：关闭时间同步后，不调用 `dmc_conti_change_speed_ratio`，只由启动前已设定的矢量速度执行。仅取消 UI 勾选但仍发送一次动态倍率命令，不能作为严格对照。

### 5.4 Qt 定时器与硬件调用抖动

**机理：** Windows 上的 `QTimer(5 ms/10 ms)` 是尽力而为，不是硬实时定时器。当前产点、喂点和反馈刷新共享 `ContiWorker` 事件循环；一次同步硬件调用或一次 Trace 批量读取过久时，后续定时任务会延迟执行。

**诊断特征：**

```text
日志中的诊断 t 间隔明显大于预期
连续多次喂点间隔异常增大
hostQueue 突然积压或突然耗尽
Trace 帧读取数量出现异常突增/缺失
```

**优化方向：** 先记录产点、喂点、Trace 读取各自的实际间隔与最长耗时；确认瓶颈后，再决定是否缩短 Trace 每次处理量、拆分上层调度职责或提高预压时间。不得仅凭感觉盲目缩短定时周期。

### 5.5 规划速度与卡侧速度能力不匹配

五次多项式零初末速度轨迹的峰值速度近似为：

```text
v_peak = 1.875 × 位移 / 轨迹时长
```

例如 `500° / 10 s` 的峰值约为 `93.75°/s`。若矢量最大速度设为 `12°/s`，则不可能在 10 s 内完成；公共倍率上限也无法弥补该差距。

这通常不直接造成“走走停停”，但会使时间同步控制长期饱和、计划时钟与实际 mark 持续失配，并增加误判缓冲状态的风险。

**检查条件：**

```text
maxVectorVelocity >= v_peak
加速/减速时间与驱动器允许加速度相容
倍率上限、负载和机械速度上限均允许该速度
```

### 5.6 驱动、机械或 EtherCAT 侧保护

**机理：** 即使连续插补队列连续，驱动也可能因报警、限位、急停输入、跟随误差、使能丢失或 EtherCAT 状态异常停止或限速。

**诊断特征：**

- 轴状态机不再为正常使能运行状态；
- 轴错误码、总线错误码或 Trace 状态异常；
- Trace 指令位置仍连续，但 Trace 实际位置不再跟随；
- 官方调试软件中也出现相同行为。

该类问题应先使用官方调试软件执行单轴长行程点位运动排除，再回到连续插补链路。

### 5.7 轨迹本身的正常起止

五次多项式在起点和终点速度为零。因此起步阶段缓慢、终点减速至零是预期行为，不应误判为顿挫。判据是：速度变化应平滑且只发生在轨迹两端；若中途周期性反复发生，才属于本文件的排查对象。

## 6. 推荐排查顺序

### 阶段 A：排除外部与基础运动问题

1. 在官方调试软件中执行单轴长行程、固定目标速度的点位运动；
2. 检查驱动报警、限位、急停链路、EtherCAT 状态和错误码；
3. 确认脉冲当量、运动方向、最大速度和加减速参数正确；
4. 确认轨迹峰值速度不超过矢量速度、驱动和机构能力。

### 阶段 B：分离连续插补基本功能与流式供给

1. 单轴参与连续插补，保持第二轴不动；
2. 关闭真正的动态倍率 API；
3. 将可容纳的短轨迹一次性预装入卡侧；
4. 比较“一条长线段”与“多条 10 ms 共线小段”。

结论判定：

| 结果 | 优先结论 |
|---|---|
| 一次性预装平滑、流式喂点顿挫 | 产点/喂点/调度或缓冲策略问题 |
| 长线段平滑、小线段顿挫 | 前瞻、段长度或矢量规划问题 |
| 两者均顿挫 | 速度/加速度参数、驱动或硬件配置问题 |
| 两者均平滑、开启倍率后顿挫 | 倍率控制律或 API 调用节奏问题 |

### 阶段 C：恢复流式供给并验证不间断性

1. 固定静态倍率；
2. 逐步提高预压时间和目标缓冲时间；
3. 记录每次产点、喂点、Trace 读取的实际间隔；
4. 将 `runState=2` 且轨迹未结束作为明确的断粮事件记录；
5. 在持续数次试验中确认 mark 从起点到终点单调连续、无中途停止。

### 阶段 D：接入时间同步公共倍率

1. 从固定倍率、充分缓冲且平滑的基线开始；
2. 限制倍率 API 最小调用间隔，并记录每次实际生效值；
3. 先仅启用缓冲保护，再启用相位修正；
4. 将倍率下限设为安全但可观察的值，避免误把极慢运动判断为停止；
5. 以 Trace 的 mark/指令/实际位置和离线时间轴评估效果，不只凭肉眼。

## 7. 每次试验的最小记录集

每次试验必须保存或在运行日志中包含：

```text
试验编号、软件版本、卡号、轴号、轨迹位移、轨迹时长、产点周期
最大矢量速度、加速时间、减速时间、S 曲线时间、前瞻开关、路径误差
预压时间、目标/低水位/危险水位、倍率开关和全部倍率参数
runState、currentMark、pushedMark、remainSpace、hostQueue
缓冲估计时间、倍率 ref/phase/buffer/cmd/applied
轴状态机、轴错误码、总线错误码、Trace 状态
Trace 同帧指令位置、实际位置及其时间基准
```

数据记录采用 1 ms Trace 采样基准；UI 曲线仅作低频观察，最终结论应基于保存的原始 Trace 数据。

## 8. 当前优先修正项

在不改变总体架构的前提下，后续代码修改应按以下顺序执行：

1. 增加“真正静态倍率”模式，关闭后绝不调用 `dmc_conti_change_speed_ratio`；
2. 将“轨迹未结束但 `runState=2`”记录为断粮/提前停止事件，而不是静默继续；
3. 增加喂点间隔、产点间隔、硬件调用耗时和低水位进入/退出的诊断日志；
4. 增加阶段一的“一次性预装”和“单长线段”对照模式；
5. 用对照结果确定前瞻参数、预压时间、目标缓冲时间与任务调度方式；
6. 最后才恢复并整定时间同步公共倍率控制。

在上述基线尚未平滑前，不应继续增大倍率控制复杂度或直接进入八轴时间同步测试。
