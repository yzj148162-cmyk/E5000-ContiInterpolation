# CDPR E5000 连续插补顿挫排查与优化设计

## 1. 目的与适用范围

本文用于排查 `conti_ctrl` 当前两轴阶段一、阶段二连续插补中出现的“运动断断续续、速度反复掉落或在段端停住”现象，并定义后续优化顺序。

本文的原则是：**先通过可复现的对照试验定位，再一次只修改一个变量**。不得在尚未区分“卡侧断粮”“小线段规划”“倍率控制”“驱动侧保护”之前，同时调整多个参数后仅凭肉眼判断结果。

本文不替代以下设计文件：

- 《CDPR_E5000_连续插补方案》：连续插补测试总体方案；
- 《CDPR_E5000_8轴插补时间同步与速度倍率控制设计》：未来八轴时间同步与公共倍率方案；
- 《CDPR_E5000_数据记录与绘图设计》：Trace 数据记录与离线评估方案。

## 2. 当前程序执行链路

当前两轴测试的轨迹与喂点链路为：

```text
五次多项式轨迹
  每 10 ms 生成一个两轴绝对位置点
          ↓
上位机轨迹队列（hostQueue）
          ↓
硬件线程待喂队列（feedQueue）
          ↓
启动前预压：默认覆盖约 200 ms 的有效线段
          ↓
dmc_conti_open_list + dmc_conti_line_unit + dmc_conti_start_list
          ↓
运行中每 5 ms 检查 mark、卡侧空槽、上位机队列并尝试补段
          ↓
控制卡连续插补缓冲区、前瞻、矢量速度/S 曲线、公共速度倍率
          ↓
EtherCAT 伺服 + Trace 指令位置/实际位置反馈
```

程序当前分为两个线程层次：

- `ContiWorker` 控制/规划线程：轨迹产点默认 10 ms，连续插补状态与倍率决策 5 ms，Trace 数据提取与反馈刷新 10 ms；
- `E5000HardwareInterface` 独占硬件线程：串行执行全部雷赛 SDK 调用，并以 5 ms 定时器自动向卡侧补段。

Trace 底层采样周期在当前 1000 us 总线配置下为 1 ms；这里的 10 ms 只是上层批量提取和刷新周期。控制线程不再直接承担卡侧定时喂点，但它调用硬件接口时仍同步等待，而且 Trace、状态读取、倍率和喂点 SDK 最终都在同一硬件线程串行执行。因此某次 SDK 调用耗时过长，仍可能同时阻塞控制线程并推迟硬件线程的下一次补段。

## 3. 先明确三个概念

### 3.1 `remainSpace` 是空槽数，不是剩余轨迹量

`dmc_conti_remain_space` 返回连续插补缓冲区中**还可以写入多少段**。

```text
remainSpace 很大（例如接近 5000） → 卡侧已排队的段很少，甚至为空
remainSpace 很小                   → 卡侧已排队的段较多，接近容量上限
```

因此，“缓冲剩余空间很大”不能视为安全；对连续运动而言，它反而可能表示卡侧即将断粮。

### 3.2 `currentMark / pushedMark` 是段进度，不是精确时间

`currentMark` 来自 `dmc_conti_read_current_mark`，表示控制卡已执行到的连续插补段进度；`pushedMark` 是程序已成功压入的最后一个 mark。

```text
待执行段数近似值 = pushedMark - currentMark
```

该值适合判断是否接近断粮，但 mark 的精确边界语义必须通过实机单段试验校准，不能将一个 mark 的偏差直接等同为固定时间误差。

### 3.3 “停止”与“顿挫”不是同一问题

- **停止**：控制卡或伺服明确进入停止、暂停、报警或失能状态；
- **顿挫**：电机仍在连续执行，但每个小段之间反复减速、倍率反复降低，或短暂等待下一批线段。

两者需要用 `runState`、mark、缓冲量、倍率和 Trace 曲线共同区分，不能只靠肉眼判断。

## 4. 当前代码中明确会停止运动的路径

当前程序没有调用 `dmc_conti_pause_list`、`dmc_conti_delay`。`dmc_conti_change_speed_ratio` 用于在线调整倍率，不是暂停接口。

但以下路径会调用 `dmc_conti_stop_list`、关闭连续插补列表，或使运动自然结束：

| 类别 | 触发条件 | 预期日志/状态 | 是否属于故障 |
|---|---|---|---|
| 用户操作 | 点击减速停止、全局立即停止、安全关闭控制卡、退出程序 | “减速停止”“立即停止”“安全停机” | 否 |
| 正常结束 | 轨迹生产结束、所有段已消费 | “五次多项式轨迹已执行完成” | 否 |
| 库函数错误 | 开表、压入线段、启动、倍率调整等调用失败 | “错误：...” | 是 |
| Trace 错误 | 连续插补运行期间 Trace 读取失败 | “运行中位置 Trace 读取失败” | 是 |
| 首段未执行 | 启动后超时，`currentMark` 未进入首个有效段 | “仍未消费首个有效段” | 是 |
| 驱动/总线侧 | 伺服报警、限位、急停链路、EtherCAT 异常 | 轴错误码、状态机、总线状态变化 | 是 |

`dmc_conti_get_run_state` 的常用含义如下：

| runState | 含义 | 当前排查含义 |
|---:|---|---|
| 0 | 运行 | 控制卡正在消费连续插补段 |
| 1 | 暂停 | 应检查是否存在外部暂停或库函数调用 |
| 2 | 正常停止 | 可能是轨迹自然结束，也可能是卡侧段已经耗尽 |
| 3 | 未启动 | 启动流程或状态异常 |
| 4 | 空闲 | 列表关闭、错误停止或未运行 |

特别注意：当前程序已将 `3/4` 视为意外停止；当轨迹生产尚未结束或硬件待喂队列仍有点时出现 `runState=2`，也会明确记录为流式断粮或卡侧提前结束并进入安全停止。若轨迹已经生产完成且硬件待喂队列为空，当前代码会接受 `runState=2` 为结束条件，即使 `currentMark < pushedMark`；这可能包含控制卡 mark 边界语义带来的一段固定偏差，仍需通过单段试验校准并在日志中保留最终 mark 差值。

## 5. 顿挫原因清单与判据

### 5.1 卡侧断粮或近断粮

**机理：** 控制卡完成已排入的最后一条线段，但后续轨迹尚未成功压入。卡只能在最后目标点减速、停住或等待，之后即使又有新点，也会形成可见的速度断裂。

**可能诱因：**

- 轨迹产点速度低于卡侧消费速度；
- 预压时间过短；
- 喂点定时器延迟；
- Trace、状态读取、倍率等 SDK 调用占用独占硬件线程，推迟其 5 ms 补段任务；
- 当前倍率高于可持续的供点速度；
- 产点因脉冲量化被跳过，导致有效段覆盖时间不足。

**诊断特征：**

```text
currentMark 接近 pushedMark
缓冲段数趋近 0
remainSpace 迅速增大、接近卡侧总容量
上位机总队列也可能接近 0
runState 可能由 0 变为 2
```

界面显示的“上位机队列”是控制线程 `hostQueue` 与硬件线程待喂队列之和；控制线程会很快把新点移交给硬件线程，因此不能只观察内部 `hostQueue` 是否为空。

**首要对照试验：** 在阶段一把整条短轨迹一次性预装入卡侧（不超过卡侧段容量），并关闭时间同步倍率控制。若顿挫消失，问题位于流式产点/喂点或其调度，而不是电机和连续插补基本功能。

### 5.2 小线段没有被前瞻平滑融合

**机理：** 当前每 10 ms 生成一条 `dmc_conti_line_unit` 小线段。若控制卡没有将相邻段充分前瞻融合，就可能在每一段端点按矢量加减速规划减速，再进入下一段加速。

**当前风险：** 已启用小线段前瞻，但 `PathError=0`。对于完全共线的阶段一轨迹，理论上不需要拐角容差；但是否实际融合、需要多少前瞻段，必须实机验证，不能仅以“已调用设置函数”判定有效。

**诊断特征：**

```text
mark 持续前进
上位机总队列或卡侧时间缓冲有余量
remainSpace 未表现为接近空队列
倍率基本稳定
但 Trace 实际位置的速度呈周期性锯齿或反复接近零
```

**首要对照试验：** 以相同起终点分别执行：

1. 一条长直线段；
2. 由当前 10 ms 小段组成的同一共线轨迹。

若只有第二种顿挫，应优先检查前瞻配置、可用前瞻段数、轨迹离散周期和矢量加减速参数。

### 5.3 公共速度倍率被压低或频繁变化

**机理：** 时间同步控制为防止断粮，会在低水位时把倍率压向下限；相位误差、参考速度和缓冲修正也会共同改变倍率。倍率不应命令零速，但若反复变化，仍会造成明显的加减速感。

**当前边界：** 倍率下限大于零时，倍率控制本身不应主动让电机完全静止；但它可能通过降低/提高消费速度，掩盖或放大供点不足问题。

**诊断特征：**

```text
“在线速度倍率已生效”日志与顿挫时间重合
ratioApplied 在较短时间内反复变化，或长期贴近 ratioMin
bufferTimeMs 经常低于 lowBufferMs / criticalBufferMs
```

**对照试验：** 当前已实现真正的“静态倍率模式”：关闭时间同步后不调用 `dmc_conti_change_speed_ratio`，只由启动前设定的矢量速度执行。试验时应同时确认日志中没有倍率 API 调用，且“倍率 API 距今”没有继续计时。

### 5.4 Qt 定时器与硬件调用抖动

**机理：** Windows 上的 `QTimer(5 ms/10 ms)` 是尽力而为，不是硬实时定时器。当前产点、状态监控和反馈刷新位于控制/规划线程，卡侧自动喂点位于独占硬件线程；但全部 SDK 操作仍在硬件线程串行执行。一次 Trace 批量读取、状态查询或其他 SDK 调用过久时，既会让同步等待的控制线程延迟，也可能推迟硬件线程的补段定时器。

**诊断特征：**

```text
日志中的诊断 t 间隔明显大于预期
连续多次喂点间隔异常增大
上位机总队列突然积压或突然耗尽
Trace 帧读取数量出现异常突增/缺失
```

**优化方向：** 先记录产点、硬件补段服务、Trace 读取各自的实际间隔与最长耗时；确认瓶颈后，再决定是否缩短 Trace 单次处理量、进一步隔离硬件任务或提高目标缓冲时间。不得仅凭感觉盲目缩短定时周期。

### 5.5 规划速度与卡侧速度能力不匹配

五次多项式零初末速度轨迹的峰值速度近似为：

```text
v_peak = 1.875 × 位移 / 轨迹时长
```

例如 `500° / 10 s` 的峰值约为 `93.75°/s`。若矢量最大速度设为 `12°/s`，则不可能在 10 s 内完成；公共倍率上限也无法弥补该差距。

为建立可重复的单轴对照基线，建议将该组试验的最大矢量速度设为 **100°/s**。它略高于五次多项式的理论峰值，可避免基础速度上限主动压缩规划速度；以后开启时间同步时，参考倍率峰值约为：

```text
ratio_ref_peak = 93.75 / 100 = 0.9375
```

因此，100°/s 也便于比较“关闭时间同步的卡侧默认执行”与“开启时间同步后的时间历程跟踪”。这里的 100°/s 是插补系允许的最大矢量速度，不表示电机会始终以 100°/s 匀速运行；测试前仍须确认驱动器、机械结构及安全限速允许该值。

这通常不直接造成“走走停停”，但会使时间同步控制长期饱和、计划时钟与实际 mark 持续失配，并增加误判缓冲状态的风险。

**检查条件：**

```text
maxVectorVelocity >= v_peak
加速/减速时间与驱动器允许加速度相容
倍率上限、负载和机械速度上限均允许该速度
```

### 5.6 驱动、机械或 EtherCAT 侧保护

**机理：** 即使连续插补队列连续，驱动也可能因报警、限位、急停输入、跟随误差、使能丢失或 EtherCAT 状态异常停止或限速。

**诊断特征：**

- 轴状态机不再为正常使能运行状态；
- 轴错误码、总线错误码或 Trace 状态异常；
- Trace 指令位置仍连续，但 Trace 实际位置不再跟随；
- 官方调试软件中也出现相同行为。

该类问题应先使用官方调试软件执行单轴长行程点位运动排除，再回到连续插补链路。

### 5.7 轨迹本身的正常起止

五次多项式在起点和终点速度为零。因此起步阶段缓慢、终点减速至零是预期行为，不应误判为顿挫。判据是：速度变化应平滑且只发生在轨迹两端；若中途周期性反复发生，才属于本文件的排查对象。

## 6. 推荐排查顺序

### 阶段 A：排除外部与基础运动问题

1. 在官方调试软件中执行单轴长行程、固定目标速度的点位运动；
2. 检查驱动报警、限位、急停链路、EtherCAT 状态和错误码；
3. 确认脉冲当量、运动方向、最大速度和加减速参数正确；
4. 确认轨迹峰值速度不超过矢量速度、驱动和机构能力。

### 阶段 B：分离连续插补基本功能与流式供给

1. 单轴参与连续插补，保持第二轴不动；
2. 关闭真正的动态倍率 API；
3. 将可容纳的短轨迹一次性预装入卡侧；
4. 比较“一条长线段”与“多条 10 ms 共线小段”。

结论判定：

| 结果 | 优先结论 |
|---|---|
| 一次性预装平滑、流式喂点顿挫 | 产点/喂点/调度或缓冲策略问题 |
| 长线段平滑、小线段顿挫 | 前瞻、段长度或矢量规划问题 |
| 两者均顿挫 | 速度/加速度参数、驱动或硬件配置问题 |
| 两者均平滑、开启倍率后顿挫 | 倍率控制律或 API 调用节奏问题 |

### 阶段 C：恢复流式供给并验证不间断性

#### C1. 建立可重复基线

先固定以下参数，每次只改变一个被测因素：

| 参数 | 基线值 |
|---|---:|
| 主动轴位移 | 500° |
| 轨迹时长 | 10 s |
| 规划周期 | 10 ms |
| 最大矢量速度 | 100°/s |
| 时间同步倍率控制 | 关闭 |
| 一次性预装全部轨迹 | 关闭 |
| 小线段前瞻 | 开启 |

加速时间、减速时间和 S 曲线时间先保持当前已验证安全的设置。时间同步关闭时，“倍率 API 距今”应处于无有效计时状态，日志中也不应出现周期性倍率调整调用，否则该次试验不能作为纯流式基线。

该静态倍率基线只用于判断流式运动是否平滑，不要求电机严格用 10 s 执行完毕；没有时间同步时，连续插补段本身不携带时间戳，卡侧会按矢量速度、加减速和前瞻结果推进。严格比较 10 s 时间历程应放在静态基线平滑之后。

#### C2. 提高启动储备并检查有效段

当前代码要求时间参数满足：

```text
产点周期 <= 启动预压时间 <= 目标缓冲时间 <= 固定执行延迟
危险水位时间 < 低水位时间
```

此外，倍率控制周期不得小于 5 ms，普通/安全倍率 API 最小间隔均不得小于倍率控制周期，并需满足 `0 < 倍率下限 <= 倍率上限 <= 2`、倍率死区不为负、单次倍率变化上限大于零。当前代码即使关闭时间同步也会统一校验这些字段；任一条件不满足都会输出同一条“测试参数不合法”日志。

固定执行延迟还不得超过轨迹能够建立的有效未来时长，否则轨迹会在达到启动条件前已经生成结束。

当前实现启动时实际向卡侧压入到“目标缓冲时间”，启动预压时间主要作为最低验收门槛；因为代码又要求目标缓冲时间不小于启动预压时间，所以单独提高启动预压、但不提高目标缓冲，并不会得到更多初始卡侧轨迹。运行中硬件线程同样持续补到目标缓冲时间。

建议先把固定执行延迟统一设为 600 ms、启动预压保持 200 ms，再将目标缓冲时间从 200 ms 依次提高到 300、400、500 ms，每轮只改变目标缓冲时间。若希望界面语义上明确测试“预压 500 ms、缓冲 500 ms”，则应使用：

```text
启动预压时间 = 500 ms
目标缓冲时间 = 500 ms
固定执行延迟 = 600 ms
```

这组数值只保证参数关系合法，并不保证 100°/s 静态倍率下长期不断粮。500°/10 s 轨迹的平均规划路径速度约为 50°/s，而关闭时间同步后卡侧可能接近 100°/s 推进；600 ms 的产点领先量可能被逐渐耗尽，尤其在五次多项式始末的实时产点位移很小时更明显。

若本轮目的只是隔离“硬件补段调度是否平滑”，应暂时建立足够大的有限未来窗口。例如保持卡侧目标缓冲 500 ms，把固定执行延迟提高到约 5000 ms，使硬件待喂队列预先持有更长的未来轨迹，但卡侧仍由硬件线程按 500 ms 目标流式补段。该值仅用于诊断，且不得超过轨迹可建立的有效未来时长；确认静态流式路径平滑后，再逐步减小执行延迟并接入时间同步，使卡侧平均消费速度与实时产点速度匹配。

时间同步关闭时，低水位和危险水位不参与倍率限速，也不决定硬件线程何时补段；它们只需保持合法关系，例如 100 ms 和 50 ms。所有时间还须受轨迹长度、卡侧实测容量和安全要求约束。

启动条件应同时检查“未来时间储备”和“有效非零段数”。五次多项式始末的位移增量很小，量化到脉冲后可能出现重复目标点；此时仅看预压时间可能显示储备充足，但卡侧并没有足够的连续有效段。

若提高启动储备后起步顿挫明显减轻，优先检查预压不足、有效段不足或首次补段不及时；若中段平滑而末段仍顿挫，则重点检查终点附近的小段量化及前瞻窗口收缩。

#### C3. 位移与时长对照

在其他设置不变的前提下，比较以下组合：

| 组别 | 位移 | 时长 | 观察目的 |
|---|---:|---:|---|
| A | 500° | 10 s | 可重复基线，理论峰值 93.75°/s |
| B | 1000° | 10 s | 增大每个规划点的位移增量 |
| C | 500° | 5 s | 增大每个规划点的位移增量 |
| D | 500° | 20 s | 减小每个规划点的位移增量 |

若 B、C 的始末顿挫减轻，而 D 更严重，就强烈支持“始末小段经脉冲量化后出现零增量或极小有效段”的判断。B、C 两组的理论峰值均为 187.5°/s，最大矢量速度仍为 100°/s 时不能用它们验证准时到达；这两组只用于在安全条件下观察段长变化对连续性的影响，必要时应选择更保守的位移/时长组合，或在确认驱动与机构安全后相应提高速度上限。

#### C4. 前瞻对照

在较低且安全的速度下，只切换小线段前瞻开关。若关闭前瞻后顿挫显著加重，说明卡侧前瞻确实在平滑小线段，但当前前瞻窗口、段长或补段方式仍不足以覆盖始末区域。

#### C5. 运行证据与判据

每次基线试验至少记录：

- 原始规划点数、有效下发段数、量化后跳过段数，并分别统计轨迹起始、中段和末段；
- 第一条有效段的规划时刻、启动时实际预压的有效段数和未来时间；
- 每次 `dmc_conti_line_unit` 调用前后的 currentMark、pushedMark、缓冲余量、单次推送段数及调用耗时；
- 每段各轴的脉冲增量，重点检查始末是否连续出现 0 pulse 或 1 pulse 小段；
- 由 Trace 指令位置差分得到的 1 ms 指令速度，以及同帧实际位置。

若 Trace 指令位置本身呈现平台—跳变，问题位于上位机下发、卡侧缓冲、脉冲量化或前瞻；若 Trace 指令连续而实际位置顿挫，再转向驱动器、总线同步和机械侧排查。还应将“轨迹仍在生产或硬件待喂队列非空时出现 `runState=2`”明确记录为断粮事件，确认 currentMark 单调前进，并记录结束时 `pushedMark-currentMark` 的差值用于校准 mark 边界语义。

#### C6. 证实原因后的流式优化方向

1. 规划线程只维护有限的未来时间窗，不预知或一次生成真实场景中的整条轨迹；
2. 硬件线程独占卡侧喂点，根据高、低水位整批补段，避免“一生成一点就调用一次硬件 API”；
3. 启动判据同时满足预压时间和有效段数；
4. 始末区域适当扩大未来规划窗口；
5. 合并量化后重复的目标点，同时保留 mark 与原规划时刻的映射；
6. 运行期间不得反复启动插补系。

在关闭时间同步的 500°/10 s、100°/s 基线尚未稳定平滑前，不进入倍率控制调参，以免流式供给顿挫与倍率调节效应叠加。

### 阶段 D：接入时间同步公共倍率

1. 从固定倍率、充分缓冲且平滑的基线开始；
2. 限制倍率 API 最小调用间隔，并记录每次实际生效值；
3. 先仅启用缓冲保护，再启用相位修正；
4. 将倍率下限设为安全但可观察的值，避免误把极慢运动判断为停止；
5. 以 Trace 的 mark/指令/实际位置和离线时间轴评估效果，不只凭肉眼。

## 7. 每次试验的最小记录集

每次试验必须保存或在运行日志中包含：

```text
试验编号、软件版本、卡号、轴号、轨迹位移、轨迹时长、产点周期
最大矢量速度、加速时间、减速时间、S 曲线时间、前瞻开关、路径误差

runState、currentMark、pushedMark、remainSpace、hostQueue
缓冲估计时间、倍率 ref/phase/buffer/cmd/applied
轴状态机、轴错误码、总线错误码、Trace 状态
Trace 同帧指令位置、实际位置及其时间基准
```

数据记录采用 1 ms Trace 采样基准；UI 曲线仅作低频观察，最终结论应基于保存的原始 Trace 数据。

## 8. 当前优先修正项

当前已经完成以下诊断基础：

- 关闭时间同步后不调用动态倍率 API；
- 将轨迹仍在生产或硬件待喂队列非空时的 `runState=2` 记录为断粮/提前停止；
- 记录 mark、缓冲时间、倍率值和硬件补段服务间隔；
- 提供“一次性预装全部轨迹”对照模式；
- 卡侧补段由独占硬件线程定时执行。

下一步按以下顺序推进：

1. 使用 500°/10 s、100°/s 静态倍率基线，先建立足以避免规划线程被追上的固定执行延迟，再分别测试目标缓冲 200、300、400、500 ms；
2. 补充产点间隔、每次 SDK 调用耗时、有效段数、最终 mark 差值和始末量化跳过点的分区统计；
3. 完成“单长线段”与“10 ms 共线小段”的对照；
4. 根据证据确定目标缓冲时间、前瞻参数及硬件线程单次补段策略；
5. 静态倍率流式基线平滑后，再恢复并整定时间同步公共倍率控制。

在上述基线尚未平滑前，不应继续增大倍率控制复杂度或直接进入八轴时间同步测试。
