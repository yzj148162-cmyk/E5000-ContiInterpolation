# CDPR E5000 Trace 位置延迟标定与 8 轴扩展设计

## 1. 目的与边界

速度模式位置闭环使用 `dmc_vmove` 启动连续运动，并通过
`dmc_change_speed_unit` 周期更新速度命令。控制器的实时误差始终是：

```text
e_control(t) = q_reference(t) - q_actual(t)
```

该误差直接服务于 PID，不能因为 Trace 反馈存在固定相位延迟而人为错移。

本标定只解决诊断问题：同一 Trace 帧中的 type05 指令位置和 type06
实际位置并不一定代表同一物理时刻。若直接相减，会把固定传输/执行延迟造成的
位置差也当作跟随误差。

第一版只自动运动一个人工选择并已使能的轴；程序启动、控制卡初始化或轴使能时
都不会自动执行标定运动。

## 2. 采样量与时间基准

标定使用控制卡同一 Trace 采样帧内的四个对象：

- type03：指令速度；
- type04：实际速度；
- type05：指令位置；
- type06：实际位置。

所有时间由 Trace 帧序号和 Trace 周期重建，不用上位机收到批量数据包的时刻。
因此上位机线程一次取回多帧不会破坏帧间时间关系。

原始 type03/04/05/06 帧仍由异步记录器完整保存，任何延迟补偿都不覆盖原始数据。

## 3. 单轴标定运动

默认速度序列为：

```text
+30, -30, +60, -60, +100, -100 °/s
```

每段流程如下：

1. 静止等待；
2. 调用速度模式启动函数进入目标速度；
3. 保持并收集 Trace 帧；
4. 调用减速停止；
5. 确认轴停止后进入下一段。

默认保持时间 1500 ms、稳定采样窗口 500 ms、段间静止 500 ms。
“单段行程上限”在启动前检查 `速度 × 保持时间`，避免错误参数造成过大位移。
正负速度成对执行，使一档速度的往返位移近似抵消，同时便于分离静态位置偏置。

## 4. 延迟拟合

每个稳定速度段计算：

```text
v_i = mean(type03)
e_i = mean(type05 - type06)
```

稳定窗口由“轴数据有效、Trace 帧序号连续、type03 已进入目标速度容差”
确定，再从最长连续区间中央截取用户设置的采样窗口。type04 的 1 ms
逐帧速度可能存在量化和微分抖动，因此不再用“每一帧实际速度都贴近指令”
作为硬筛选条件；程序改为检查整个窗口内的实际速度均值，并记录均值、标准差
供诊断。这样既能确认电机确实跟上目标速度，也不会因为单帧速度噪声反复切断
本来有效的稳态窗口。

六组数据使用最小二乘拟合：

```text
e_i = τ v_i + b
```

其中：

- `τ`：有效位置延迟；
- `b`：与速度方向无关的静态位置偏置；
- `R²`：线性模型对六组数据的解释程度。

通过条件：

- Trace 无丢帧；
- `0 ms ≤ τ ≤ 20 ms`；
- `R² ≥ 0.98`；
- 三组正反向独立延迟相对总拟合结果的最大离散不超过 1.5 ms。

标定不通过时不覆盖原有有效值。若该轴没有有效历史结果，则继续采用官方软件
测得的 8 ms 默认值，并在结果表中明确标记来源。

每一段结束后，运行日志和 `events.log` 都记录目标速度、捕获帧数及序号范围。
分析阶段逐段输出有效轴帧、指令稳定帧、最长连续区间、实际选用帧、段内丢帧、
指令/实际速度均值、实际速度标准差及位置差均值。若无法拟合，日志必须直接
给出失败段和具体原因，不得把未计算的结果显示成 `τ=0、R²=0`。

## 5. 诊断误差补偿

对当前实际位置帧 `t`，在 type05 历史中查询 `t-τ`：

```text
e_comp(t) = q05(t - τ) - q06(t)
```

当 `τ` 不是 Trace 周期的整数倍时，在相邻两帧 type05 之间做线性插值。
Trace 重配、帧序号跳变或历史不足时，本帧补偿结果标记为无效；界面显示
“等待历史帧”，不会伪造补偿值。

以下数据同时保留：

- 原始同帧误差 `q05(t)-q06(t)`；
- 延迟对齐后的指令位置；
- 延迟补偿诊断误差；
- 使用的延迟值及其来源。

速度闭环 PID、最大跟随误差保护和速度前馈仍使用规划参考与实际位置，
不使用上述诊断补偿量。

## 6. 结果保存与失效

每个轴独立保存：

- 应用延迟与实测延迟；
- 静态偏置；
- `R²`、RMSE、正反向离散和丢帧数；
- 来源、状态和标定时间。

保存文件位于系统应用数据目录下的
`CDPR/E5000ContiInterpolation/trace_delay_calibration.json`。

程序启动后加载结果。若当前 EtherCAT 总线周期或 Trace 周期与保存时不同，
所有实测结果失效并回退为默认 8 ms，必须重新标定。

## 7. 当前 UI 操作流程

1. 初始化控制卡；
2. 在全局轴状态区使能待标定轴；
3. 打开“Trace延迟标定”页签；
4. 选择轴、板卡 unit 和安全运动参数；
5. 确认六段运动不会超出机械行程；
6. 点击“开始单轴标定”；
7. 观察速度曲线、进度和运行日志；
8. 完成后查看八轴结果表及 `e=τv+b` 拟合图。

“减速停止”“立即停止”、全局急停、全局失能和关闭程序都会终止当前标定。

## 8. 后续 8 轴扩展

结果结构和 UI 表从第一版起就保留 8 个轴，但自动运动仍严格限制为单轴。
扩展时建议采用逐轴标定，而不是八轴同时运动：

1. 检查所有在线轴、使能状态和机械安全范围；
2. 固定其他七轴，仅对当前轴执行六段标定；
3. 每轴独立拟合、验收和保存；
4. 单轴失败时停止序列，不覆盖该轴原结果；
5. 全部完成后生成 8 轴延迟向量；
6. 多轴运行时按各轴自己的 `τ_i` 对齐 type05，再与同帧 type06 比较。

当前 `TraceTelemetryFrame` 仍沿用阶段 A 的双轴载荷；真正接入 8 轴同帧诊断前，
还需把帧内固定双轴数组扩展为 8 轴，并同步核对控制卡单次 Trace 对象数量和
上位机批量读取吞吐量。

未来若增加“自动标定全部轴”，必须额外实现逐轴确认、绳索张力/行程保护、
失败回滚和人工授权，不能把当前单轴按钮简单改成循环八次。
