# CDPR E5000 8 轴插补时间同步与速度倍率控制设计

## 1. 目的与适用范围

本文定义未来 CDPR 8 轴连续插补阶段的**时间同步与公共速度倍率控制**方案。

目标是在雷赛 E5000 连续插补“线段无时间戳”的约束下，使控制卡的轨迹消费进度尽量跟随上位机规划的时间历程，同时保证连续插补缓冲区不会断粮。

本文适用于以下未来链路：

```text
10 ms 动力学/状态计算
        ↓
每个 10 ms 段细分为 10 个 1 ms 末端轨迹点
        ↓
逆运动学得到 8 轴绝对目标位置 Q[i]
        ↓
上位机延迟轨迹队列（默认约 50 ms）
        ↓
E5000 连续插补缓冲区 + 前瞻 + 公共速度倍率
        ↓
Trace 指令/实际位置监控
```

本文**不改变当前 `conti_ctrl` 阶段一、阶段二的实现**。当前测试程序每 10 ms 只生成一个两轴五次多项式点，速度倍率仅在启动时设置一次；尚未生成 1 ms 八轴点，也尚未启用本文的闭环倍率控制。

---

## 2. 问题定义

### 2.1 连续插补的时间边界

`dmc_conti_line_unit` 接收的是一条多轴直线段的终点位置，不携带“必须在某个绝对时刻到达”的时间戳。控制卡仍会依据：

- 矢量速度与加减速参数；
- S 曲线参数；
- 前瞻结果；
- 当前速度倍率；
- 后续缓冲线段；

规划并执行运动。

因此，连续插补可以保证多轴沿给定线段序列协调运动，但不能天然保证第 `i` 个点严格在第 `i × 1 ms` 到达。

### 2.2 本方案的目标

本方案不试图将连续插补伪装成严格 PVT。目标应表述为：

1. 8 轴保持同一轴空间轨迹和同步比例；
2. 控制卡执行进度相对于规划时钟的误差保持有界；
3. 板卡缓冲区保持安全余量，避免断粮自动结束；
4. 倍率变化平滑，避免可见的速度振荡；
5. 用 Trace 区分“卡侧轨迹进度”和“电机真实跟随状态”。

对于视觉连续、自然的 CDPR 运动，此方案应能达到良好效果；对于每个 1 ms 点都必须严格按时到达的需求，应采用带时间戳的 PVT 或更底层的周期同步控制。

## 3. 为什么使用公共速度倍率

8 轴轨迹可写为：

```text
Q(s) = [q0(s), q1(s), ..., q7(s)]
```

其中 `s` 是整条轨迹的公共推进参数。速度倍率改变的是 `s(t)` 的推进速度，即整条八维轨迹同时变快或变慢。

```text
公共倍率升高：所有轴按原比例更快通过同一条 Q(s)
公共倍率降低：所有轴按原比例更慢通过同一条 Q(s)
```

这不会改变 8 轴之间的插补比例，也不会改变理想路径几何形状。

不应为每个轴分别用倍率追赶。各轴独立调速会改变轴间比例，导致实际位置偏离规划的八维线段；单轴能力不足、跟随误差或报警应由轴限速、Trace 监控和安全策略处理。

## 4. 时间、mark 与缓冲量定义

### 4.1 轨迹点和 mark

未来轨迹以 `dt = 1 ms` 离散。第 `i` 个轨迹点包含 8 轴绝对目标位置：

```text
Q[i].pos[8]
Q[i].planTime = i × dt
Q[i].mark = 递增编号
```

每次调用 `dmc_conti_line_unit` 压入一条线段时，附带该线段的递增 `mark`。控制卡通过 `dmc_conti_read_current_mark` 返回当前执行进度。

`mark` 的边界语义（“当前执行段”还是“最近完成段”）必须在实机用递增单段测试校准。以下公式允许存在一个固定的 `markOffset`，避免把该实现细节误判为时间误差。

### 4.2 固定执行延迟

为避免 Windows 调度抖动造成卡侧断粮，规划点不会一生成就立刻执行。建议先建立固定执行延迟：

```text
delay = 50 ms
```

含义是：某个点在上位机生成后，至少在约 50 ms 后才允许由板卡执行。该延迟由上位机延迟队列和板卡缓冲余量共同形成。

### 4.3 应执行 mark

设：

- `tExecuteStart`：首个轨迹点的预定执行起始时刻；
- `firstMark`：该首点对应的 mark；
- `dt`：规划点周期，未来为 1 ms；
- `now`：单调时钟当前时刻。

则理论执行进度为：

```text
elapsed = max(0, now - tExecuteStart)
markExpect = firstMark + floor(elapsed / dt) + markOffset
```

应将 `markExpect` 限制在本次轨迹的 `[firstMark, lastMark]` 范围内。

### 4.4 相位误差与缓冲余量

```text
phaseErrorMark = markExpect - currentMark
bufferSegments = pushedMark - currentMark
bufferTimeMs ≈ bufferSegments × dtMs
```

其中：

- `phaseErrorMark > 0`：卡侧进度落后于时间表；
- `phaseErrorMark < 0`：卡侧进度超前于时间表；
- `bufferSegments`：已压入但尚未执行的线段数量估计。

`phaseErrorMark` 与 `bufferSegments` 必须同时使用：前者反映时间同步，后者反映安全余量。

## 5. 三层速度倍率控制

### 5.1 总体结构

在正常缓冲区范围内，倍率命令可以表示为：

```text
ratioCmdRaw = ratioRef + ratioPhase + ratioBuffer
ratioCmd = clamp(ratioCmdRaw, ratioMin, ratioMax)
```

三个量均为无量纲倍率或倍率修正量。实际代码中可采用乘法结构或 PI 结构，但必须保留以下职责划分：

| 分量 | 作用 | 主要输入 |
|---|---|---|
| `ratioRef` | 使平均执行速度接近规划时间历程 | 近期轨迹点间距、规划点周期、矢量速度基准 |
| `ratioPhase` | 消除提前/滞后的时间相位误差 | `markExpect - currentMark` |
| `ratioBuffer` | 维持安全缓冲余量 | `pushedMark - currentMark` |

安全策略优先级高于上述求和：当缓冲余量进入危险区时，必须允许安全限速覆盖“追赶时间”的要求。

### 5.2 `ratioRef`：轨迹速度前馈

对近期 `N` 条规划线段（建议 `N = 5~20`）计算八维矢量长度：

```text
L[i] = norm(Q[i + 1].pos - Q[i].pos)
Vref = mean(L[i] / dt)
```

若 `Vbase` 为连续插补设置的矢量速度基准，则可采用：

```text
ratioRef = clamp(Vref / Vbase, ratioMin, ratioMax)
```

此量是前馈项，反映规划轨迹本身近期需要快走还是慢走。实际使用时需要考虑加减速和 S 曲线约束；若 `Vref` 超出卡侧可实现范围，应限幅并记录“规划速度不可达”。

### 5.3 `ratioPhase`：时间相位校正

在缓冲安全范围内，根据 `phaseErrorMark` 生成小幅修正：

```text
ratioPhase = clamp(Kphase × phaseErrorMark,
                   -phaseCorrectionLimit,
                   +phaseCorrectionLimit)
```

解释：

- 卡侧落后：`phaseErrorMark > 0`，允许小幅提高倍率；
- 卡侧超前：`phaseErrorMark < 0`，允许小幅降低倍率；
- `Kphase` 应很小，目标是渐进纠偏而不是一次性追平；
- 必须限幅，避免因为一次 mark 抖动造成倍率突变。

### 5.4 `ratioBuffer`：缓冲水位修正与安全覆盖

设定三个水位：

```text
bufferLow    = 30 ms
bufferTarget = 40 ms
bufferHigh   = 50 ms
```

在 `bufferLow < bufferTimeMs < bufferHigh` 的正常范围内，`ratioBuffer` 可为很小的比例修正，帮助水位回归目标值。

危险区必须采用优先级覆盖：

```text
if bufferTimeMs <= bufferLow:
    ratioCmd = min(ratioCmd, ratioSafeLow)
    // 不允许为了追赶时间而加速消耗剩余缓冲。

if bufferTimeMs <= bufferCritical:
    请求减速停止或立即停止

if bufferTimeMs >= bufferHigh:
    可逐步解除保守限速；是否主动提高倍率仍需受相位误差和最大速度限制。
```

缓冲水位控制的首要目的不是严格校时，而是防止连续插补断粮。

## 6. 例子

假设：

```text
dt = 1 ms
当前按时间表应执行 mark = 120
currentMark = 116
pushedMark = 155
ratioRef = 0.70
```

则：

```text
phaseErrorMark = 120 - 116 = +4 ms
bufferTimeMs ≈ 155 - 116 = 39 ms
```

39 ms 位于 30~50 ms 的安全范围内，卡侧略落后。因此可采用：

```text
ratioPhase  = +0.03
ratioBuffer =  0.00
ratioCmd    =  0.73
```

卡会略微加速，逐渐减小 4 ms 的相位误差。

若 `pushedMark = 140`，则缓冲仅约 24 ms。即使卡侧落后，也不得提高倍率；应优先降低倍率并等待补点恢复余量。

## 7. 更新周期、死区与滤波

### 7.1 更新周期

建议在缓冲监控/补点线程中每 `5~10 ms` 计算一次倍率。该周期不等于每次都调用 `dmc_conti_change_speed_ratio`。

### 7.2 倍率变化死区

记录最后一次已写入卡的倍率 `ratioApplied`，建议：

```text
ratioDeadband = 0.01

if abs(ratioCmd - ratioApplied) < ratioDeadband:
    不调用倍率调整 API
else:
    dmc_conti_change_speed_ratio(card, crd, ratioCmd)
    ratioApplied = ratioCmd
```

当缓冲进入低水位或危险区时，应绕过死区，立即执行保守限速或停止策略。

### 7.3 平滑滤波和限速

建议对已限幅的命令做一阶平滑：

```text
ratioFiltered = alpha × ratioApplied + (1 - alpha) × ratioCmd
```

初始可取 `alpha = 0.8~0.95`。同时设置单次变化上限，例如每个监控周期不超过 `0.02~0.05`。这些参数必须通过实机测试整定，不能仅凭公式固定。

## 8. 与小线段前瞻的关系

小线段前瞻用于查看后续线段并规划更连续的段间速度；公共速度倍率用于整体改变这条已规划路径的推进速度。

```text
前瞻：决定相邻小线段如何平滑连接
倍率：决定整条已规划路径整体走快还是走慢
```

前瞻通常会改善视觉平滑性，但也意味着各 mark 的实际经过时刻不是严格固定的。因此本方案使用 mark 相位误差做闭环校正，而不假设“每条线段天然正好执行 1 ms”。

前瞻需要足够的未来线段。未来 1 ms 方案建议板卡缓冲目标保持约 40~50 条小段；当前阶段一/二默认 8 条、且每条代表 10 ms 生成点，仅用于基础功能验证。

## 9. Trace 反馈与时间误差的分层

必须区分三层量：

| 层级 | 数据来源 | 用途 |
|---|---|---|
| 卡侧轨迹消费进度 | `currentMark` | 判断插补队列相对于计划时钟是否提前/落后 |
| 卡侧指令位置 | Trace type 5 | 判断控制卡实际输出的轴目标与规划位置是否一致 |
| 电机实际位置 | Trace type 6 | 判断伺服跟随误差与机械实际运动状态 |

`currentMark` 只能说明连续插补执行到了哪一段附近，不能证明电机已经无误差到达该轨迹点。

建议记录并分析：

```text
hostTimestamp
markExpect
currentMark
phaseErrorMark
pushedMark
bufferSegments / bufferTimeMs
ratioRef / ratioPhase / ratioBuffer / ratioApplied
Trace command position[8]
Trace actual position[8]
following error[8]
```

Trace 读取仍应采用当前项目已使用的 Trace PDO 同帧 type 5（指令位置）与 type 6（实际位置），不使用普通位置读取 API 计算实时跟随误差。

## 10. 未来软件模块划分

未来在当前架构中增加下列职责，不改变“硬件 API 由 `E5000HardwareInterface` 独占线程访问”的原则：

```text
ContiWorker / 轨迹层
├── 10 ms 状态计算与轨迹段生成
├── 1 ms 轨迹插值
├── 8 轴逆运动学与 HostTrajectoryQueue
├── 计划时钟与 markExpect 计算
└── 时间同步控制器（生成 ratioCmd）

E5000HardwareInterface / 硬件层
├── 连续插补开表、压点、启动、停止
├── 读取 currentMark / remainSpace / runState
├── 调用 dmc_conti_change_speed_ratio
└── Trace PDO 读取与缓存
```

建议将倍率控制器实现为无硬件调用的纯逻辑模块，例如：

```cpp
struct RatioControlInput
{
    qint64 nowNs;
    long firstMark;
    long lastMark;
    long pushedMark;
    long currentMark;
    double referenceVectorSpeed;
    double bufferTimeMs;
};

struct RatioControlOutput
{
    double ratioRef;
    double ratioPhase;
    double ratioBuffer;
    double ratioCommand;
    bool safetyOverride;
};
```

硬件线程仅在输出超过死区或安全覆盖触发时调用 `dmc_conti_change_speed_ratio`。

## 11. 建议初始参数

| 参数 | 初始建议 | 说明 |
|---|---:|---|
| 动力学计算周期 | 10 ms | 未来可根据性能缩短 |
| 电机轨迹点周期 `dt` | 1 ms | 10 ms 段细分为 10 点 |
| 固定执行延迟 | 50 ms | 优先保证缓冲余量 |
| 板卡目标余量 | 40 ms | 约 40 条 1 ms 小段 |
| 板卡低水位 | 30 ms | 低于此值禁止追赶加速 |
| 板卡高水位 | 50 ms | 高于此值可逐步解除保守限速 |
| 倍率范围 | 0.2~1.0 | 需服从轴能力和实机验证 |
| 倍率更新周期 | 5~10 ms | 与补点监控同级或更慢 |
| 倍率死区 | 0.01 | 防止频繁来回写倍率 |
| 相位修正窗口 | 5~20 点 | 用于平滑估计参考速度 |
| `PathError` | 0 | 初期优先验证路径不偏离 |

上述数值仅为首轮实机测试起点。若实际线段执行时间明显不是 1 ms，必须先根据 Trace 和 mark 数据重新标定，再调整控制参数。

## 12. 8 轴测试顺序与验收指标

### 12.1 建议顺序

1. 8 轴静态配置、使能和 Trace PDO 验证；
2. 8 轴共线小位移连续插补，固定倍率运行；
3. 验证 `currentMark`、`pushedMark`、`remainSpace` 的语义与稳定性；
4. 建立 1 ms 点和约 50 ms 延迟队列；
5. 仅启用 `ratioRef`；
6. 加入小幅 `ratioPhase`，观察时间相位误差是否收敛且无振荡；
7. 加入缓冲水位安全覆盖；
8. 在转向、曲线、不同绳长变化率和外力扰动下验证；
9. 对比 Trace 指令/实际位置，建立跟随误差与安全阈值。

### 12.2 首轮验收指标

```text
1. 连续插补不因缓冲区空而自动结束；
2. pushedMark - currentMark 长期位于目标水位附近；
3. ratioApplied 不出现高频来回跳变；
4. phaseErrorMark 在设定容忍窗口内有界，不长期单向累积；
5. 8 轴 Trace 指令位置与规划 Q[i] 的空间关系正确；
6. Trace 实际位置跟随误差不超过安全阈值；
7. 未出现异常振动、啸叫、过速、张力尖峰或轴报警。
```

## 13. 关键限制

1. 公共速度倍率只能做整体时间缩放，不能补偿单轴独立误差；
2. 已压入板卡的连续插补线段不应频繁修改；
3. 前瞻、S 曲线和加减速会使单段耗时偏离理想 `dt`；
4. `currentMark` 是插补进度，不是电机真实到位证明；
5. 任何“追赶时间”的策略都不得突破缓冲余量、轴速度、加速度、张力和安全限位约束；
6. 若时间精度要求提升到每个 1 ms 点严格到达，本方案不应作为最终保证手段。
