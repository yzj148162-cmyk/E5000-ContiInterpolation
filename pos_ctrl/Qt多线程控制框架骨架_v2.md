# Qt/C++ 多线程控制框架骨架（聚焦 UI 粗略显示 + 规划线程 + 硬件线程）

## 1. 目标与范围

本文档聚焦三个核心线程的工程骨架设计：

- **UI线程**：粗略显示、参数输入、启停操作
- **规划线程**：根据当前反馈和轨迹目标，生成下一周期或一段时间的轨迹点
- **硬件线程**：严格周期执行，负责 EtherCAT/PDO 交互与轨迹点消费

目标是先建立一个**最小可运行、可扩展、职责清晰**的 Qt/C++ 框架，为后续增加动力学补偿、力分配、串口接收等线程打基础。

---

## 2. 线程划分与职责

| 线程 | 实现方式 | 周期 | 核心任务 |
|---|---|---:|---|
| 硬件线程 | 继承 `QThread`，重载 `run()` | `1ms`（与总线周期同步） | 发送 PDO、读取电机/传感器反馈、消费轨迹点队列、触发规划线程 |
| 规划线程 | 继承 `QThread`，重载 `run()` | 跟随硬件周期（条件变量唤醒） | 读取反馈，计算下个周期/下一段目标位置，写入轨迹队列 |
| UI线程 | Qt 主线程 + `QTimer` | `50~100ms` | 显示状态、接收用户轨迹点、下发给硬件线程 |

### 2.1 UI线程职责

UI线程只负责**非实时工作**：

- 参数输入
- 启停按钮
- 状态粗略显示
- 曲线/文本刷新
- 将用户输入的轨迹点或目标命令送入共享队列

UI线程**不直接参与 1ms 周期控制**，不直接进行 PDO 读写。

### 2.2 规划线程职责

规划线程负责根据当前状态生成控制目标：

- 读取最新反馈
- 计算下一周期目标位置，或下一小段轨迹点
- 写入轨迹点队列
- 被硬件线程按周期唤醒

规划线程应尽量做到：

- 与硬件线程解耦
- 不在硬件线程中做重计算
- 支持后续扩展为五次多项式、补偿、前馈等算法模块

### 2.3 硬件线程职责

硬件线程是**唯一严格周期线程**：

- 使用 `sleep_until` + `steady_clock` 锁定 1ms 周期
- 每周期完成：
  - 读取轨迹点
  - 发送 PDO
  - 读取反馈
  - 写入共享反馈结构
  - 通知规划线程计算下一周期内容

原则上，所有与 EtherCAT 周期强相关的动作都放在这里。

## 3. 痛点与解决方案

| 痛点 | 解决方案 |
|---|---|
| UI 与硬件线程数据交换，避免阻塞 | 使用**共享结构体 + `std::mutex`**。硬件线程写反馈（加锁）；UI 定时器加锁复制后刷新。UI 下发轨迹点：加锁推入队列。 |
| 硬件线程需要严格 1ms 周期 | 使用 `std::this_thread::sleep_until` 配合 `std::chrono::steady_clock`，避免使用 `sleep(1)`。必要时提高线程优先级。 |
| 规划线程需要与硬件周期同步 | 硬件线程每周期结束后，通过 `std::condition_variable` 唤醒规划线程；规划线程计算下个周期目标。 |
| 多个实时线程（后续扩展）的数据竞争 | 每个共享数据结构配独立互斥锁（反馈锁、轨迹锁、补偿锁等），锁内只做简单赋值/拷贝。 |
| 担心互斥锁成为 1ms 周期瓶颈 | 保持锁内代码极短。优先用普通互斥锁即可，后续仅在实测确有瓶颈时再升级为双缓冲/无锁队列。 |
| `QThread` 选择 `run()` 还是 `moveToThread` | **实时周期性控制**用重载 `run()`；**事件驱动型任务**（如串口接收）用 `moveToThread + Worker`。当前硬件/规划线程用 `run()`。 |
| 硬件线程如何安全传递数据给 UI | 方案1：发送信号（Qt 自动排队到主线程）。方案2：共享数据 + UI 定时器读取。当前推荐**共享数据 + UI 定时器粗略刷新**。 |

---

## 4. 关键机制与数据提示

### 4.1 精确周期等待

```cpp
std::this_thread::sleep_until(start_time + 1ms);
```

其中：

```cpp
start_time = std::chrono::steady_clock::now();
```

说明：

- 使用 `steady_clock`，避免系统时间跳变影响周期
- 使用 `sleep_until`，比简单 `sleep_for(1ms)` 更适合做周期控制

### 4.2 条件变量同步

- 硬件线程：`cv.notify_all();`
- 规划线程：`cv.wait(lock, []{ return flag; });`

注意：

- 唤醒标志位要及时重置
- 规划线程不能空转忙等
- 用条件变量做“按周期唤醒”

### 4.3 互斥锁保护共享数据

使用：

- `std::lock_guard<std::mutex>`
- 或 `std::unique_lock<std::mutex>`（用于条件变量）

原则：

- 锁内只做结构体赋值、浅拷贝、队列 `push/pop`
- 锁外做复杂计算

### 4.4 线程安全队列

可先用：

```cpp
std::queue<T>
```

配合互斥锁保护，用于存放轨迹点。

建议：

- UI线程 `push`
- 规划线程 `push`
- 硬件线程 `pop`

如果后期发现吞吐或争用成为问题，再替换为环形缓冲或无锁队列。

### 4.5 停止线程

每个线程循环都检查：

```cpp
std::atomic<bool> running
```

主线程停止流程：

1. 设置 `running = false`
2. 调用 `cv.notify_all()`，避免线程卡在等待态
3. 调用 `wait()` 等待线程退出

### 4.6 UI定时器

UI 刷新使用：

```cpp
QTimer::start(50);
```

在槽函数中：

1. 加锁复制共享状态
2. 释放锁
3. 更新控件/曲线

避免在 UI 刷新时直接长时间持锁。

## 5. 建议的共享数据结构

### 5.1 反馈数据

```cpp
struct FeedbackData {
    int32_t actualPosRaw = 0;
    int32_t targetPosRaw = 0;
    uint16_t statusWord = 0;
    int8_t modeDisplay = 0;
    int32_t errorRaw = 0;
    bool running = false;
    bool fault = false;
};
```

用途：

- 硬件线程每周期写入
- UI线程定时读取显示
- 规划线程读取用于计算下一周期目标

### 5.2 轨迹点

```cpp
struct TrajectoryPoint {
    int32_t targetPosRaw = 0;
    int32_t targetVelRaw = 0;   // 可选
    int32_t targetAccRaw = 0;   // 可选
};
```

用途：

- UI或规划线程生成
- 硬件线程消费

### 5.3 同步控制对象

```cpp
struct SyncContext {
    std::mutex feedbackMutex;
    std::mutex trajMutex;
    std::condition_variable cv;
    bool plannerWakeup = false;
};
```

## 7. 推荐的最小运行流程

### 7.1 启动阶段

1. 主线程初始化界面
2. 初始化共享上下文
3. 创建并启动硬件线程
4. 创建并启动规划线程
5. 启动 UI 定时器

### 7.2 运行阶段

#### 硬件线程循环

每 1ms：

1. 获取轨迹点
2. 发送 PDO
3. 读取反馈
4. 写入共享反馈
5. 唤醒规划线程
6. `sleep_until` 到下个周期

#### 规划线程循环

被唤醒后：

1. 读取反馈
2. 计算下一周期/下一段目标
3. 写入轨迹点队列
4. 继续等待下次唤醒

#### UI线程

每 50~100ms：

1. 读取共享反馈
2. 刷新显示
3. 接收用户输入
4. 向轨迹队列写入点或命令

## 8. 后续扩展建议

### 8.1 增加动力学/阻抗/力分配线程

模式与规划线程一致：

- 等待硬件周期或事件唤醒
- 读取反馈和规划目标
- 计算补偿量
- 写入共享补偿结构
- 硬件线程在发送 PDO 前叠加补偿

### 8.2 增加事件驱动线程

例如：

- 串口接收
- UDP/TCP 接收
- 上位机通讯

这些线程更适合：

- `moveToThread + Worker`
- 事件循环驱动

如果回调阻塞严重，再考虑 `run()`。

### 8.3 性能优化顺序

不要一开始就过度优化。推荐顺序：

1. 先使用普通互斥锁
2. 实测周期抖动和锁争用
3. 仅在确有必要时再升级为：
   - 原子变量
   - 双缓冲
   - 无锁队列

## 9. 一句话总结

**硬件线程重载 `run()` 严格锁住 1ms；规划线程被条件变量唤醒；UI线程定时粗略读取共享数据。**

先实现这个最小骨架，再逐步扩展动力学、补偿、通讯等功能，保持架构一致即可。

## 10. 关键注意事项（新增）

### 10.1 关于“软件周期与总线周期对齐”

- 不需要、也无法做到**软件线程与控制卡内部 EtherCAT 时钟“严格锁相”**
- 正确目标是：
  - 总线周期 = 1ms
  - 伺服周期 = 1ms
  - 硬件线程周期 ≈ 稳定 1ms
- 只要三者**同频且稳定**，系统即可正常运行
- 不要尝试“卡边沿”或补偿时钟偏差

👉 核心原则：
**做“同频稳定”，不做“锁相对齐”。**

---

### 10.2 关于 OP 状态与伺服使能

- **OP（Operational） ≠ 伺服使能（Operation Enabled）**
  - OP：EtherCAT通信进入运行态
  - 使能：驱动器状态机进入可运动状态

- 两种控制方式（二选一）：

#### 方式A：使用雷赛封装
- 使用 `nmc_set_axis_enable`
- 轮询状态机直到 Operation Enabled

#### 方式B：手动 CiA402 控制（当前推荐）
- 通过 PDO 写控制字：
  - `0x6040: 06 → 07 → 0F`
- 读取 `0x6041` 判断状态

👉 重要原则：
**不要混用两种方式**

---

### 10.3 当前推荐策略

在 CSP + PDO 直控架构下：

- 使用 **手动 0x6040 控制状态机**
- 全部状态控制由软件统一管理
- 避免使用 `nmc_set_axis_enable` 与之混用

---

## 11. 总结补充

- 周期问题：只需保证**1ms稳定运行**
- 状态机问题：选择一种方式**统一控制**
- 架构问题：保持线程职责清晰，不混层
