# 软件架构介绍

## 1. 文档目的

本文档用于说明当前 Qt 多线程运动控制程序的软件架构，重点回答以下问题：

- 工程为什么按当前目录结构拆分
- 各模块分别负责什么
- 三个核心线程之间如何协作
- 轨迹规划、补队列、点下发的执行链是什么
- 当前时间基准和关键状态量分别代表什么

本文档基于当前代码实现整理，适合作为后续维护、交接和继续扩展功能时的快速入口。

---

## 2. 总体架构概览

当前程序采用“UI 线程 + 规划线程 + 硬件线程”的三线程架构：

- UI 线程：负责参数输入、按钮操作、状态显示、日志显示
- 规划线程：负责轨迹生成、低水位补队列
- 硬件线程：负责 1ms 周期执行、轨迹点消费、控制卡写入、反馈回读

整体数据流如下：

1. 用户在 UI 上输入运动参数并点击“开始轨迹”
2. UI 线程写入一份 `MotionRequest`
3. 规划线程读取请求，重建轨迹
4. 规划线程将精细轨迹点按批次补入共享队列
5. 硬件线程按 1ms 周期从队列中消费一个点并下发给控制卡
6. 硬件线程回读反馈并写入共享上下文
7. UI 刷新控制器按较慢周期读取共享快照并显示到界面

## 4. 三个核心线程如何协作

### 4.1 UI 线程

UI 线程在程序中最先被用户感知，但它不负责实时控制。它主要完成：

- 参数采集
- 命令下发
- 状态显示
- 日志显示

它通过 `SharedContext` 把运动请求写给规划线程，通过 `HardwareCommand` 把控制命令发给硬件线程。

### 4.2 规划线程

规划线程是“生产者”。

它的核心特点是：

- 启动后常驻
- 大部分时间睡眠
- 根据当前任务的唤醒策略被事件或周期唤醒
- 对不同输出类型生产不同形态的结果

当前代码中的唤醒策略已经按任务类型区分：

- 当前实现中，`PVT(PVTS)`：事件唤醒
- 当前实现中，`CSP`：周期唤醒 + 低水位补队列

这只是当前模式到抽象策略的映射，不代表未来所有“周期唤醒”任务都叫 `CSP`。

所以规划线程不再统一理解为“每 1ms 都参与工作”，而是一个更通用的上层结果生成线程。

### 4.3 硬件线程

硬件线程是“消费者”和“实时执行者”。

它的核心特点是：

- 程序启动后即存在
- 严格按 1ms 周期循环
- 只有在 `motionActive_ == true` 时才真正下发轨迹点
- 每周期读取反馈并更新共享快照

这意味着：

- 硬件线程的存在周期，不等于运动执行周期
- 真正开始下发轨迹点，是在点击“开始轨迹”并完成起步准备之后

## 6. 轨迹规划与下发执行链

### 6.1 开始轨迹时的执行顺序

点击“开始轨迹”后，系统执行顺序如下：

1. UI 线程采集当前参数，形成 `MotionConfig`
2. UI 线程写入新的 `MotionRequest`
3. UI 线程向硬件线程投递 `StartMotion`
4. 硬件线程完成模式配置、轴使能等准备动作
5. 对于 `CSP` 这类队列供货型任务，硬件线程会先清空共享队列中的存量旧点
6. 硬件线程主动唤醒规划线程，并等待首批轨迹点入队
7. 规划线程检测到新请求后，重建轨迹
8. 规划线程在真正补队列前，会再次确认当前请求仍与本轮规划一致；若检测到已切换到新请求，则丢弃旧批次，不再入队
9. 规划线程向共享队列补入首批 1ms 精细点
10. 硬件线程确认首批点已到位后，才正式进入 `motionActive_`
11. 硬件线程每 1ms 消费一个新点并下发

这套逻辑的关键改进点是：

- 不再依赖“延时 20ms 大概率够用”这种经验性起步方式
- 改成显式等待首批点入队
- 对于队列供货型输出，新增“启动前清旧队列 + 入队前校验当前请求”两道轻量保护，避免新一轮 `CSP` 误用上一轮残留点

### 6.2 规划线程如何补队列

规划线程每次被唤醒后：

1. 读取 `MotionRequest`
2. 若检测到新任务，则重建轨迹
3. 根据 `PlanningOutputType` 决定当前任务属于哪种输出形态
4. 再按对应策略决定是否补队列、整表预装或写入最新控制量

相关系统参数：

- `plannerBatchSize`
- `plannerQueueLowWatermark`

这使得规划线程形成“低水位补货”机制。

需要注意的是：

- 当前实现中，这套低水位补队列机制用于 `CSP` 模式
- 但从架构定义上说，它适用于所有“上层按系统规划周期产出未来一段 1ms 序列”的任务
- 因此“队列供货型”不能被理解成 `CSP` 的别名
- 当前代码还额外加入了“入队前请求一致性校验”：
  - 如果规划线程已经算出一批点，但此时 `MotionRequest` 已切换到新任务
  - 则这批旧点会被直接丢弃，不再重新写回共享队列

对于 `PVT(PVTS)` 模式：

1. 规划线程先按系统规划周期重建整条轨迹表
2. 但不会把轨迹拆成批次放入共享队列
3. 而是把完整轨迹表写入共享上下文
4. 供硬件线程在开始阶段一次性整表下发给控制卡

至于后续的阻抗、力分配等扩展，当前代码只在共享上下文中预留了 `latestCommand` 槽位，
用于未来接入“实时控制量型”输出；本次代码暂未将其真正接入执行链。

### 6.3 精细轨迹点如何生成

当前 `QuinticPlanner` 逻辑如下：

1. 按系统规划周期生成五次多项式粗点
2. 每个粗点包含时间、位置、速度
3. `CSP` 模式下，在相邻粗点之间使用段内插值恢复 1ms 精细点
4. `PVT(PVTS)` 模式下，直接保留系统规划周期粗点作为整表点列
5. 将当前模式对应的整条轨迹先缓存到内部
6. 当前实现中，`CSP` 按批次吐给共享队列
7. 当前实现中，`PVT(PVTS)` 直接作为整表下发的数据源

当前精细插值方法为：

- Cubic Hermite 插值

### 6.4 最后一批点如何收口

当前逻辑已经做了比早期版本更严谨的收口：

1. 规划线程在内部缓存彻底取空后，将 `plannerFinished` 标记为完成
2. 硬件线程运行中如果发现：
   - 当前周期没有取到新点
   - `plannerFinished == true`
   - 共享队列也为空
3. 则认为最后一批点已经消费完毕
4. 硬件线程自动结束本次运动

这样做的目的，是避免：

- 队列已经空了，但仍然无限重复发送最后一个点

对于 `PVT(PVTS)` 模式，收口逻辑不同：

1. 硬件线程在开始阶段将整条系统规划周期轨迹表一次性通过 `dmc_pvts_table_unit` 装入控制卡
2. 再通过 `dmc_pvt_move` 一次性启动运动
3. 运行期间硬件线程不再逐点下发，而是轮询当前轴的 PVT 运行状态
4. 当控制卡报告运动完成后，硬件线程自动结束本次运动

## 7. 当前状态量语义

### 7.1 已发送轨迹点数

来源：

- `SharedContext::sentPointCount`

含义：

- 本次运动中，成功下发到控制卡的新轨迹点数

计数规则：

- 点击“开始轨迹”时清零
- 只有“从队列取到一个新点并且写卡成功”才加一
- 点击“停止轨迹”后停止增加

这比早期“硬件线程周期计数”更符合运动执行语义。

### 7.2 规划线程周期计数

来源：

- `SharedContext::queueFillCount`

含义：

- 规划线程实际向共享队列补入一批轨迹点的次数

因此它具有两个明确特征：

- 当前实现中，`CSP` 模式下，只有真正补了一批点才会增加
- `PVT(PVTS)` 模式下，由于不走补队列链路，因此该值保持不变

如果后续新增的某个模式同样采用“队列供货型”输出，那么它也应复用这套计数语义。

### 7.3 队列深度

来源：

- `trajectoryQueue.size()`

含义：

- 当前已经补好、但尚未被硬件线程消费的精细轨迹点个数

### 7.4 plannerFinished

来源：

- `SharedContext::plannerFinished`

含义：

- 当前请求对应的规划器内部缓存是否已经完全输出完毕

它主要用于：

- 硬件线程判断最后一批点消费完后是否可以自动收口

## 8. 使能、失能与运动控制关系

当前代码中已经不再使用手写 `CiA402` 控制字使能序列，而改为直接使用雷赛库函数：

- `nmc_set_axis_enable`
- `nmc_set_axis_disable`

对应封装位于：

- [hardware/EthercatInterface.cpp](C:/Users/DELL/Desktop/pos_ctrl/hardware/EthercatInterface.cpp)


开始轨迹时，不同模式的执行链如下。

### 8.1 CSP 模式

1. 规划线程生成整条 1ms 精细轨迹
2. 硬件线程配置 CSP 模式参数
3. 硬件线程调用雷赛库函数使能轴
4. 硬件线程显式等待首批轨迹点进入共享队列
5. 硬件线程进入 `motionActive_`
6. 硬件线程每 1ms 从队列消费一个点，并通过 PDO 写入目标位置

这里的 `CSP` 只是当前采用“队列供货型 + 周期唤醒”这一组合的一种具体实现。

### 8.2 PVT(PVTS) 模式

根据《雷赛控制EtherCAT总线卡用户使用手册20231027.pdf》第 7.20.2.2 节，`PVTS` 模式只需要：

- 理想轨迹上各数据点的位置
- 各数据点的时间
- 整条轨迹的起点速度
- 整条轨迹的终点速度

当前程序中的 `PVT` 模式即按这套 `PVTS` 逻辑实现：

1. 规划线程按系统规划周期生成整条轨迹表
2. 硬件线程等待完整轨迹表就绪
3. 硬件线程将每个表点的“时间、位置”整理成数组
4. 将轨迹首点速度和末点速度作为整段边界速度
5. 调用 `dmc_pvts_table_unit` 一次性装载该轴整条轨迹表
6. 调用 `dmc_pvt_move` 一次性启动运动
7. 运行期间硬件线程只监测，不再逐点下发

同时，`PVT(PVTS)` 模式在运行期间不再每 1ms 周期唤醒规划线程。
规划线程只在“开始新任务”这一类事件边界被唤醒，用于准备整表轨迹。

失能时的行为是：

1. 若当前在运动，则先停止当前运动
2. 再调用雷赛库函数失能轴

## 9. 当前架构的优点

### 9.1 职责边界清晰

- UI 不直接碰硬件
- 规划器不负责 1ms 调度
- 硬件线程不负责生成轨迹

### 9.2 适合继续扩展

后续如果需要加入：

- 动力学计算
- 阻抗控制
- 力分配
- 在线修正
- 更复杂的规划器

只需要扩展对应层，而不必重写整个程序骨架。

### 9.3 时间基准明确

- 上层算法跑在系统规划周期
- 下层执行统一细化到 1ms
- UI 显示周期与控制周期分离

### 9.4 起步与收口更稳定

- 起步阶段显式等待首批点
- 收尾阶段显式判断最后一批点是否已消费完成

### 9.5 唤醒策略更符合模式语义

- 当前实现中，`CSP` 在运行中持续补队列，因此映射为周期唤醒
- 当前实现中，`PVT(PVTS)` 在启动后只监测，因此映射为事件唤醒
- 代码中已把“输出类型”和“唤醒策略”抽成独立概念，减少了模式判断散落在各个线程中的情况

## 10. 当前实现的边界与后续可继续加强的点

虽然当前架构已经较清晰，但仍有一些后续可以继续优化的方向：

- 当前 `QuinticPlanner` 仍然是“先生成整条精细轨迹缓存，再分批搬运”，还不是真正的在线滚动生成
- 误差曲线横轴当前近似使用“已发送点数 × 1ms”，后续可进一步和实际运动时间轴更严格对齐
- `PVT` 现已按 `PVTS` 方式接入为“整表下发一次启动”的单轴实现；后续如果要扩展到多轴同步 PVT，还需要补齐多轴整表组织、轴列表同步启动和更细的运行监测
- `latestCommand` 共享槽位已预留，但“实时控制量型”模式尚未真正接入执行链
- 本次只把“周期内展开器”思路写入文档，代码中尚未单独实现该模块
- 当前低水位、批大小还是固定参数，后续可根据规划耗时和硬件节拍进一步整定
- 后续扩展模式若同样按系统规划周期产出未来 1ms 序列，应归入“队列供货型”理解，而不应因为功能名字不同就被简单归入“实时控制量型”

### 10.1 周期内展开器（设计思路，暂未写入代码）

对于后续的动力学、阻抗、力分配等扩展，当 `系统规划周期 = n * 硬件周期` 时，
不建议笼统地把两个规划周期之间的处理都称为“插值”。

更合理的思路是引入一个独立的“周期内展开器”概念，用于把慢周期结果展开成 1ms 可执行指令。

建议按结果类型区分处理方式：

- 轨迹量
  - 使用轨迹插值，例如分段多项式、Hermite
- 控制量
  - 使用保持、限幅、变化率限制、必要时一阶平滑
- 参数量
  - 使用离散更新或渐变调度

当前代码中还没有单独实现这个模块，但后续接入实时控制量型任务时，建议优先把这层抽出来，而不是把所有周期内处理逻辑直接塞进硬件线程。

## 12. 一句话总结

当前工程的核心设计思想是：

- UI 线程只负责交互与显示
- 规划线程负责按输出类型生成上层结果
- 硬件线程只负责按 1ms 周期执行轨迹点并访问控制卡
- 三者通过共享上下文和线程安全队列解耦

这套结构已经具备了继续向“整表预装型 + 队列供货型 + 实时控制量型”统一框架扩展的基础。
