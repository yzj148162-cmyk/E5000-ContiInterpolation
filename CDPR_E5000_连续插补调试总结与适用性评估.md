# CDPR E5000 连续插补调试总结与适用性评估

## 1. 文档目的

本文整理 E5000 连续插补测试程序从基础功能搭建、单轴验证、流式补点、时间同步控制到顿挫排查的完整过程，并回答以下问题：

1. 当前连续插补方案是否能够满足 CDPR 的 8 轴实时、同步、带时间历程控制需求；
2. 已出现的问题主要来自程序实现、参数设置，还是运动模式本身与需求之间的差异；
3. 哪些工作可以继续复用，后续运动执行通道应如何选择。

本文结论限定于当前项目需求、E5000 板卡、现有固件及已验证的连续插补 API 行为，不将结论泛化为“连续插补不适合所有机器人”或“E5000 无法用于任何多轴运动”。

## 2. 结论摘要

### 2.1 主要结论

基于目前的实机试验，**不建议继续将 E5000 连续插补作为本项目 8 轴 CDPR 实时控制的主执行通道**。

最关键的证据是：

- 整条已知轨迹一次性预装后，单轴可以连续、平滑运行；
- 改为符合真实需求的“在线产点 + 流式追加”后，即使卡侧没有明显断粮、上位机补点及时、时间同步倍率关闭，仍反复出现规律性顿挫、规划边界停止或已接收但暂不执行的现象；
- 增大预压和缓冲只能降低部分顿挫频率，不能从根本上消除问题；
- 将规划、反馈、硬件喂点拆分到不同线程后，流式问题仍然存在，说明它不只是 Qt 定时器或同线程调度阻塞造成的；
- Trace 中“指令位置/指令速度”与实际运动同时停止，表明主要问题已经出现在控制卡生成的运动指令层，而不是伺服单纯跟随不上；
- 某些流式试验中，新增小线段已经被 API 接收，但直到轨迹全部提交并关闭插补列表后，剩余段才连续执行，说明控制卡内部前瞻、规划或段提交机制并不等价于本项目所需的确定性在线点流；
- 连续插补段本身没有时间戳，公共速度倍率只能整体改变插补坐标系的进给速度，无法直接保证“第 k 个 8 轴位置必须在指定绝对时刻到达”。

因此，当前模式更适合：

- 整条几何路径预先已知；
- 控制目标以轮廓连续和路径进给为主；
- 允许控制卡根据前瞻、加减速和 S 曲线重新安排各段执行时间。

而本项目需要的是：

- 轨迹点由动力学和反馈在线生成；
- 8 根绳的目标位置具有统一且明确的时间语义；
- 新点需要按确定周期持续接入；
- 执行延迟、实际进度和反馈采样之间必须可计算、可验证。

二者在控制目标和数据接口上存在结构性差异。

### 2.2 建议决策

1. 停止继续围绕连续插补进行大量补丁式优化，不再把它作为阶段三 8 轴控制的默认方案；
2. 保留现有连续插补程序，作为离线已知轨迹、板卡性能及多轴几何联动的对照测试工具；
3. 下一阶段优先评估带明确时间参数的多轴 PVT 或其他时间驱动接口；若仍坚持不采用 CSP，则必须确认候选接口具有在线追加、统一时基和可定义的欠载行为；
4. 现有基础配置、硬件线程、Trace、数据记录、绘图、安全停机和 UI 可以继续复用，只替换运动执行后端。

### 2.3 试验范围说明

目前最充分的实机证据来自阶段一单轴连续插补。阶段二的双轴时间同步和阶段三的 8 轴 CDPR 控制没有继续扩大测试，因为单轴已经无法稳定满足“未知整轨迹、在线追加、按时间执行”的最低准入条件。该停止点属于先完成底层能力验证、再决定是否扩轴，而不是用单轴结果直接推断所有多轴现象。

## 3. 项目原始需求与验收标准

### 3.1 最终控制对象

最终对象是 8 绳 6 自由度 CDPR。8 个电机位置不是 8 条互不相关的轨迹，而是同一末端状态在同一时刻对应的 8 个绳长/卷筒位置。

未来轨迹可能由动力学正解、反馈状态和控制算法实时计算，整条轨迹通常不能在运动开始前一次性确定。

### 3.2 对运动接口的实际要求

理想的运动执行接口至少应满足：

- 8 轴共享同一硬件时间基准；
- 每组 8 轴目标位置具有明确的目标时刻或固定采样周期；
- 支持运动过程中持续追加未来目标点；
- 在线追加不会触发不可预测的停止、重新规划或长期等待；
- 缓冲不足、数据中断和列表结束的行为有明确契约；
- 可以读取与物理执行进度一致的同步状态；
- 可以通过 Trace/PDO 获取同一采样时刻的指令位置和实际位置；
- 上位机绘图、日志和 UI 不阻塞硬件运动。

### 3.3 连续插补接口的核心差别

当前连续插补接口接收的是一系列几何线段端点，并结合以下参数生成路径运动：

- 最大矢量速度；
- 加速、减速和停止速度；
- S 曲线时间；
- 小线段前瞻；
- 插补坐标系公共速度倍率。

端点只描述“走到哪里”，没有直接描述“必须在什么时刻到达”。因此，连续插补天然解决的是几何路径和进给问题，而不是时间索引的多轴位置控制问题。

## 4. 已完成的软件和测试基础

本次工作并非只有连续插补试验。以下基础模块已经完成并经过不同程度的实机验证：

- 独立的 `conti_ctrl` Qt C++ 工程；
- 控制卡初始化、关闭、轴使能、失能、减速停止和立即停止；
- E5000/EtherCAT 基础配置和总线周期设置、读回；
- 轴脉冲当量配置；
- 单轴相对/绝对点位测试和软件零位；
- 与官方调用顺序一致的点位运动调用链；
- 独占硬件访问线程，避免 UI 直接并发调用控制卡；
- 规划线程与硬件喂点线程的分层；
- Trace 同步采样：指令位置 type 5、实际位置 type 6；
- 1 ms 总线周期和 1 ms Trace 采样周期；
- 五次多项式测试轨迹和等间距直线对照轨迹；
- 两轴 Trace 异步记录与低频曲线显示；
- 运行状态、mark、缓冲余量、矢量速度、Trace 速度和补点调度日志；
- 一次性预装与实时流式两种对照模式；
- 公共速度倍率时间同步试验框架。

这些模块仍可为后续 PVT 或其他执行接口服务。

## 5. 调试过程回顾

### 5.1 基础轴配置和轴数问题

最初程序按 0～7 轴统一配置，但现场只连接少量 EtherCAT 轴，导致对未接入轴调用 `dmc_set_equiv` 返回错误。之后改为只配置 UI 中实际参与测试的轴。

结论：该问题属于测试轴范围配置，不是连续插补模式的根本问题。

### 5.2 位置反馈改为 Trace 同步采样

普通位置读取函数不能满足实时跟随误差测量要求，因此参照原工程和 `trace_slave_read_extract`，改用 Trace 同步读取：

- type 5：指令位置；
- type 6：实际位置。

Trace 后续成为判断顿挫来源的关键依据：当电机顿挫或停止时，卡侧指令速度本身也下降到 0，而不是仅有实际位置偏离指令位置。

结论：顿挫不能简单归因于伺服跟随误差或机械负载；至少在已记录试验中，控制卡输出给伺服的指令轨迹已经发生停顿。

### 5.3 单轴点位运动问题修正

点位运动早期出现目标方向不正确、目标小于当前位置却仍向正方向运动等问题。对照官方上位机流程后，统一采用：

1. 设置轴速度曲线；
2. 设置 S 曲线；
3. 按相对或绝对模式调用点位运动函数。

修正后，单轴位移和方向均能正确执行。

结论：电机、驱动、轴方向、脉冲当量和基础点位控制链路具备正常工作的条件；连续插补顿挫不是“电机根本无法正确运动”。

### 5.4 连续插补基本功能接通

程序完成了插补坐标系、矢量速度、加减速、S 曲线、前瞻、mark、启动、停止和缓冲监控等调用。

动态倍率函数曾在插补尚未进入可调速状态时返回 5049。之后将倍率调用延后，并在纯流式对照试验中完全关闭时间同步倍率，固定倍率为 1.0。

结论：后续“倍率关闭仍顿挫”的试验排除了动态倍率 API 频繁调用是全部顿挫的唯一原因。

### 5.5 一次性预装整条轨迹：运动平滑

勾选“一次性预装全部轨迹”后，规划轨迹在启动前完整写入卡侧，单轴运动可以连续完成，明显顿挫消失。

该试验说明：

- E5000 连续插补基本功能可以工作；
- 伺服、EtherCAT 和基础运动参数可以实现平滑运动；
- 直线端点和单位换算总体正确；
- 当控制卡在启动前获得完整未来路径时，前瞻和速度规划具有充分条件。

但该模式不能满足最终需求，因为真实 CDPR 控制中未来轨迹由上位机在线计算，不能事先获得整条路径。

### 5.6 实时流式追加：出现规律性顿挫

不预装整条轨迹时，程序先预压一定数量的段，然后在运动过程中持续追加新段。此模式下出现了起止顿挫、周期性顿挫或运行到规划边界后等待的现象。

对照试验中使用过：

- 五次多项式非等距轨迹；
- 等间距直线轨迹；
- 时间同步倍率开启和关闭；
- 500 ms 和 2000 ms 等不同预压/目标缓冲；
- 不同脉冲当量和 card unit 定义；
- 单点追加和批量追加。

等间距轨迹且倍率固定 1.0 时仍会顿挫，说明问题不能只归因于五次多项式始末段过短，或倍率控制追踪不稳定。

### 5.7 增大预压和缓冲：只能改变顿挫节奏

在等间距对照轨迹中，假设：

- 每段位移为 0.5°；
- 最大矢量速度为 30°/s；
- 500 ms 缓冲对应约 50 段；
- 2000 ms 缓冲对应约 200 段。

若卡侧以 30°/s 执行，则：

```text
50 × 0.5° / 30°/s ≈ 0.83 s
200 × 0.5° / 30°/s ≈ 3.33 s
```

这里的 0.83 s 和 3.33 s 是相应轨迹段在电机端的理论执行时间，不是写入 API 的耗时。

实机上增大预压/缓冲后，顿挫次数减少或间隔增大，但没有消失。这与“控制卡按一批未来段完成一轮规划，运动到新的规划边界后又等待后续规划”的现象相符。

结论：更大的缓冲可以掩盖问题，但代价是更长执行延迟；它没有建立稳定的在线流式执行契约。

### 5.8 线程解耦：排除了单纯的 UI/同线程阻塞

为避免规划、反馈和卡侧 API 相互阻塞，程序进行了分层：

- 规划/产点负责按模拟实时周期生成下一目标点；
- 硬件线程独占 E5000 API，负责向卡侧写入；
- Trace、数据记录和低频绘图与控制调用解耦。

日志中常见硬件补段调度间隔为数百微秒到数毫秒，卡侧剩余空间也没有显示持续耗尽。线程拆分后，一次性预装仍平滑，但实时流式仍存在顿挫。

结论：线程解耦是正确的软件架构改进，但它没有解决控制卡内部对流式小线段的规划和释放问题。因此，“上位机线程调度是唯一根因”的假设不成立。

### 5.9 card unit 和重复点过滤试验

为排除上位机数值量化造成的重复点，程序增加了三组板卡单位定义：

| `dmc_set_equiv` | card unit 含义 | 物理角度换算 |
| --- | --- | --- |
| 500.622 pulse/unit | 1 unit = 1° | Trace 仍按 500.622 pulse/deg |
| 50.0622 pulse/unit | 1 unit = 0.1° | 下发 card unit 放大 10 倍 |
| 5.00622 pulse/unit | 1 unit = 0.01° | 下发 card unit 放大 100 倍 |

同时移除了用 `llround(targetUnit * pulsesPerUnit)` 在上位机提前判定重复点的做法，只过滤真正数值重复的目标位置。

不同 unit 定义可以改变 API 参数的数值尺度，但不会改变同一物理轨迹。试验后，流式启动和停顿问题仍然存在。

结论：单位分辨率值得保留为诊断手段，但不是当前流式不连续的主因。

### 5.10 立即追加单点：API 接收不等于进入连续执行计划

程序曾尝试让规划线程每产生一个新点，硬件线程就立即调用 `dmc_conti_line_unit`，避免等待 `currentMark` 推进后再集中补段。

部分试验中出现：

- 插补已启动；
- `runState` 显示运动状态；
- `pushedMark` 持续增加；
- 缓冲空槽在写入时短暂变化；
- 但 `currentMark` 长期保持 0；
- 板卡当前矢量速度、Trace 指令速度和实际速度均为 0。

这说明 API 返回成功、点进入控制卡前端缓冲，并不必然代表该段已经被内部规划器提交为可连续执行的运动。

### 5.11 启动门控试验：初始段运动后仍停在规划边界

为避免启动过程中持续追加影响首段规划，又增加了门控策略：

1. 只写入初始预压段；
2. 等待检测到首段开始运动；
3. 再释放启动期间积累的实时点；
4. 之后继续逐点追加。

一组典型日志表现为：

- 初始写入约 51 个有效段；
- 首段运动确认后，一次释放约 71 个积累点；
- `pushedMark` 随新点持续增加；
- `currentMark` 运行到约 51 后停止；
- 板卡矢量速度和 Trace 指令/实际速度下降为 0；
- 上位机继续提交后续点，API 仍返回成功；
- 当 10 s 规划全部产生、调用关闭列表后，剩余轨迹才又连续执行。

关闭列表后，`currentMark` 又很快跳到最终 mark，但 Trace 实际位置仍持续运动较长时间才到终点。这进一步说明该 mark 在当前前瞻/列表结束流程中不能直接当作物理执行位置或时间进度。

该现象与曲线中“规划期望先到终点，实际位置长时间延迟后再走完”的结果一致。

这是当前最有说服力的反例：卡侧并非因为没有收到点而停止，而是已接收的在线追加段没有按预期及时转入连续执行计划。

### 5.12 主动批量写入试验：改变主机策略仍未根治

最新方案按卡侧前瞻段数进行主动批量写入，不再等待 `currentMark` 触发补点，希望一次提供足够的未来段供前瞻规划。

实机结果仍未解决平滑问题。

这说明继续调整“每批 1、10、25、50 或 200 段”只能改变控制卡获得未来路径的形态，无法从上位机侧建立一个已被文档和实机共同证明的确定性规则：何时新增段一定会被纳入当前运动、何时会重新规划、何时必须关闭列表。

## 6. 已排除或已弱化的原因

| 候选原因 | 对照证据 | 当前判断 |
| --- | --- | --- |
| 电机或驱动无法平滑运行 | 点位运动正常；整轨迹预装平滑 | 不是主因 |
| 轴方向或单位换算错误 | 点位方向已修正；终点物理位置正确 | 不是主因 |
| 单纯伺服跟随误差 | Trace 指令速度本身也停顿 | 不是主因 |
| 动态倍率 API 频繁调用 | 倍率控制关闭、倍率固定 1.0 时仍顿挫 | 不是唯一主因 |
| 五次多项式始末小段 | 等间距直线仍出现均匀顿挫 | 不是唯一主因 |
| 上位机重复点过滤 | 移除脉冲取整过滤并放大 card unit 后仍存在 | 不是主因 |
| 卡侧缓冲完全断粮 | 多组日志中点持续写入、空槽充足；且部分停止时 `pushedMark` 仍增加 | 不能解释全部现象 |
| UI 绘图或日志阻塞硬件 | 硬件独占线程和异步记录已接入，问题仍在 | 不是主因 |
| 预压数量过少 | 从 500 ms 增至 2000 ms 仅改善频率，未根治 | 只能缓解 |

仍然无法完全从公开信息中确认的是 E5000 内部前瞻规划、在线追加段的提交边界和 `close_list` 的具体触发语义。但这项不确定性本身就是风险：主执行接口若依赖不可观测、不可预测的内部状态，就不适合承担 8 轴实时同步控制。

## 7. 为什么该模式与 CDPR 需求不匹配

### 7.1 路径几何同步不等于时间同步

连续插补可以让多个轴沿同一个几何段协调到达段终点，但段执行时间由矢量速度、加减速、S 曲线和前瞻共同决定。

CDPR 的要求是第 k 组绳长目标对应第 k 个物理时刻。只保证几何路径不散，并不能保证该组位置在规定时刻到达。

### 7.2 轨迹段没有时间戳

当前 `dmc_conti_line_unit` 形式没有为每个端点携带到达时间。上位机只能通过公共速度倍率间接改变整条插补路径的推进速度。

这相当于对控制卡内部时间重参数化结果做外环修正，而不是直接下发带时间约束的轨迹。

### 7.3 公共倍率不能独立修正 8 个轴

公共速度倍率作用于整个插补坐标系。它适合保持几何路径比例，但无法分别处理：

- 各绳轴不同的速度、加速度和负载限制；
- 某一轴单独落后或超前；
- 8 轴不同的跟随误差；
- 单轴约束触发后其余轴如何保持原时间表。

### 7.4 前瞻需要未来路径，实时控制需要最新反馈

小线段平滑依赖控制卡提前看到足够多的未来段。未来段越多，控制卡越容易生成连续速度；但 CDPR 未来轨迹需要根据最新反馈在线计算，过早生成和锁定大量未来点会降低反馈的时效性。

因此存在直接矛盾：

- 增大预压和执行延迟：更利于前瞻和平滑，但控制使用的是更旧的状态；
- 减小预压和执行延迟：更符合实时反馈，但更容易遇到规划边界和流式提交问题。

### 7.5 `mark` 不能作为稳定的物理时间进度

设计阶段曾尝试用：

```text
相位误差 = 时间表应执行 mark - 当前执行 mark
```

再通过公共倍率追赶时间表。

实机表明 `currentMark` 可能：

- 长时间不变；
- 按批跳变；
- 与已经被 API 接收的 `pushedMark` 明显脱节；
- 在轨迹末端不能直接等价于电机已经停止。

因此它适合做段级诊断，但不能被当作 1 ms 或 10 ms 的连续物理时钟。

### 7.6 速度倍率无法构成严格的到时控制

理论上的倍率控制由轨迹参考速度、相位误差和缓冲水位共同产生：

```text
ratioCmd = ratioRef + ratioPhase + ratioBuffer
```

即使算法本身正确，仍受以下限制：

- 参考速度是上位机对规划路径的估计，不是控制卡最终采用的瞬时速度；
- 相位反馈依赖 `currentMark`，而 mark 不连续且存在内部规划延迟；
- 倍率只改变公共进给，不能指定某段确切结束时刻；
- 前瞻、加减速和 S 曲线会继续改变段间速度；
- 缓冲安全控制可能覆盖时间同步控制。

因此该方法可以用于“让整体时间肉眼上接近计划”，但不能提供 8 轴机器人控制所需的确定到时保证。

### 7.7 运行行为依赖内部规划边界

实机中最不利的行为不是简单的 API 调用耗时，而是控制卡对新增段何时纳入当前运动缺少可验证的确定性。

如果新增段必须积累到某个内部阈值、等待一次规划提交，甚至在某些情形下等待列表关闭，那么上位机无法仅靠 1 ms/10 ms 定时喂点保证连续执行。

## 8. CNC 连续插补与 CDPR 实时控制的目标差异

| 对比项 | CNC/轮廓连续插补 | 本项目 CDPR 实时控制 |
| --- | --- | --- |
| 主要目标 | 几何轮廓连续、加工进给平滑 | 8 轴在统一时刻到达指定位置 |
| 未来路径 | 通常提前已知较长路径 | 由动力学与反馈在线生成 |
| 时间语义 | 进给速度决定路径时间 | 每个状态点属于明确物理时刻 |
| 前瞻 | 越多通常越利于平滑 | 过多会增加延迟、削弱反馈时效 |
| 调速方式 | 公共进给倍率合理 | 需要时间驱动且可能涉及轴级约束 |
| 列表结束 | 已知程序自然结束 | 在线控制可能长时间没有“整轨迹结束” |
| 进度反馈 | 段号可用于加工流程 | 需要与采样时钟严格对应的执行进度 |

这并不表示 CNC 插补技术本身有问题，而是它优化的目标与本项目的控制目标不同。

## 9. 继续使用连续插补的工程风险

若继续将其推进到 8 轴，现有单轴问题会被进一步放大：

1. 单轴尚不能稳定证明流式平滑，8 轴只会增加段长度、轴约束和前瞻复杂度；
2. 公共倍率难以区分路径整体落后与某一绳轴单独落后；
3. 更长缓冲会使动力学反馈和执行之间形成更大延迟；
4. 内部规划状态不可观测，异常时难以判断是断粮、等待提交、前瞻重算还是轴约束限速；
5. 即使肉眼看起来平滑，也难以证明 8 轴轨迹的时间误差满足要求；
6. 后续正运动学复现末端轨迹时，时间轴的不确定会直接影响规划轨迹对比；
7. 大量时间可能继续消耗在调整批大小、预压深度和触发时机，而这些参数没有形成可移植的控制保证。

## 10. 建议的替代路线

### 10.1 首选：验证带时间参数的多轴 PVT

若 E5000 支持满足要求的多轴 PVT，应优先做最小化验证。重点不是函数名称，而是确认：

- 每个点是否包含位置、速度和明确时间参数；
- 时间参数作用于单段持续时间还是绝对时间；
- 多轴是否共享同一段时间；
- 是否支持运动中持续追加；
- 追加点何时生效是否有明确规则；
- 欠载时是保持、减速还是报错；
- 1 ms 或目标周期是否在支持范围内；
- 是否能读取与时间节点一致的执行进度。

如果这些条件不能得到手册、厂家和实机三方面确认，则不应仅因接口名为 PVT 就直接采用。

### 10.2 备选：选择面向周期位置控制的执行接口或控制器

若项目继续明确不使用 CSP，应寻找其他等价的时间驱动、多轴同步接口；必要时重新评估运动控制硬件。选择依据应是：

- 统一硬件时钟；
- 周期性同步接收 8 轴目标；
- 确定的缓冲和欠载行为；
- 支持在线反馈闭环；
- 能在目标周期内稳定运行；
- 指令和反馈可以按同一时间戳记录。

### 10.3 保留连续插补的合理用途

现有连续插补模式仍可用于：

- 已知整条轨迹的一次性预装测试；
- E5000 多轴几何联动能力验证；
- 与新执行接口进行离线轨迹对照；
- 非实时、无需严格到时的演示运动；
- 厂家进一步说明在线追加规则后的专项复核。

## 11. 下一阶段建议验证流程

### 11.1 先向厂家确认接口契约

建议将以下问题直接提交雷赛技术支持，并要求给出 E5000 对应固件和库版本下的明确答复：

1. 连续插补启动后，`dmc_conti_line_unit` 新增的段何时进入当前可执行计划；
2. 小线段前瞻是否按批次提交，提交阈值是什么；
3. `dmc_conti_close_list` 是否会触发剩余段的统一规划或释放；
4. `currentMark` 在前瞻、暂停、减速和列表结束时的确切语义；
5. 官方是否支持“未知终点、长期在线追加”的连续插补使用方式；
6. 是否有带段时间或点时间的多轴接口，以及推荐的最小周期。

### 11.2 对替代接口做最小闭环试验

建议按以下顺序验收：

1. 单轴：已知位置—时间曲线，验证到达时间、速度连续性和 Trace；
2. 双轴：不同位移但相同时间节点，验证同步性；
3. 在线追加：只预知有限未来窗口，持续产点，验证无整轨迹关闭也能连续运行；
4. 欠载试验：主动中断产点，确认安全且可预测的行为；
5. 8 轴：记录全部指令/实际位置时间历程，进行 CDPR 正运动学后处理。

### 11.3 建议的硬性通过条件

- 单轴在目标周期下连续运行，不依赖整轨迹预装；
- 双轴到达同一时间节点的误差可测且满足阈值；
- 流式追加过程中不因内部批次边界出现周期性停顿；
- 执行进度可映射到客观时间，而不是仅依赖跳变的段号；
- 轨迹结束、主动停止和数据欠载三种状态可明确区分；
- 指令位置与实际位置均可按统一时间轴记录。

## 12. 可复用成果与软件重构建议

### 12.1 可直接复用的模块

- UI 的全局控制区、测试页和状态显示；
- 控制卡初始化、关闭、轴使能/失能和安全停机；
- EtherCAT 周期设置与读回；
- 脉冲当量和物理单位换算；
- Trace 指令/实际位置同步读取；
- 硬件独占线程；
- 规划与硬件执行解耦；
- 异步数据记录和低频绘图；
- 五次多项式规划器；
- 运行日志和诊断指标框架。

### 12.2 建议的软件边界

后续不必推倒重写整个程序。建议抽象统一的运动执行后端，例如：

```text
轨迹/动力学规划器
        ↓  带时间的8轴目标
上位机有限未来队列
        ↓
IMotionBackend
        ├─ E5000ContiBackend（保留作对照）
        └─ PVT或其他时间驱动后端（新主路径）
        ↓
Trace/同步反馈 → 数据记录与后处理
```

这样可以保留大部分已经完成的工程工作，同时避免连续插补专用状态继续渗透到规划、UI 和数据记录层。

## 13. 汇报用最终判断

本次调试已经证明 E5000 连续插补能够执行预先完整给定的几何轨迹，但尚不能证明它能够以本项目要求的方式，稳定执行未知终点、在线产生、具有严格时间历程的多轴目标点流。

试验中最关键的差异不是“电机能不能转”，而是：

- 整轨迹预装时平滑；
- 在线流式时，已接收的未来段可能在内部规划边界处等待；
- 增大预压只能以增加延迟换取部分平滑；
- 公共倍率和 mark 无法把无时间戳几何段可靠地恢复成严格时间轨迹。

因此，继续把该模式扩展到 8 轴，将承担更高的软件复杂度和不可验证的时间同步风险。当前更合理的工程决策是：**保留连续插补作为离线对照能力，停止将其作为 CDPR 实时控制主方案，转向具有明确时间语义和在线追加契约的多轴运动接口。**
